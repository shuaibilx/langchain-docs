# Configuration

> 使用 config.toml、hooks 和 MCP 服务器配置 Deep Agents Code

Deep Agents Code 将其配置存储在 `~/.deepagents/` 目录中。主要的配置文件有：

| 文件 | 格式 | 用途 |
|------|------|------|
| `config.toml` | TOML | 模型默认值、提供商设置、构造函数参数、配置文件覆盖、主题、更新设置、MCP 信任存储 |
| `.env` | Dotenv | 全局 API 密钥和 secrets |
| `hooks.json` | JSON | 外部工具订阅 Deep Agents Code 生命周期事件 |
| `.mcp.json` | JSON | 全局 MCP 服务器定义 |

> `~/.deepagents/.state/` 下的文件保存每台机器的 Deep Agents Code 状态，由系统自动管理。

## 提供商凭证

对于每个提供商，Deep Agents Code 按顺序检查两个凭证来源：

1. **存储的密钥** — 通过 `/auth` 输入并本地持久化。
2. **环境变量** — `OPENAI_API_KEY`、`ANTHROPIC_API_KEY`、`GOOGLE_API_KEY` 等，加上带有 [`DEEPAGENTS_CODE_`](#deepagents_code_-prefix) 前缀的同名变量，以及从 [`.env` 文件](#environment-variables)加载的任何密钥。

对于同一提供商，存储的密钥始终优先于环境变量密钥。

### 使用 `/auth`（推荐）

从任何会话打开凭证管理器：

```
/auth
```

管理器列出 Deep Agents Code 知道的每个 LLM 提供商，标记已有密钥的提供商，并允许你内联粘贴、轮换或清除凭证。每个提供商显示为以下之一：

| 标签 | 含义 |
|------|------|
| `✓ credentials set` | 存储的或环境变量密钥可用 |
| `! missing OPENAI_API_KEY` | 命名的环境变量未设置且未存储密钥——选择该行粘贴一个 |
| `local provider` | 无需凭证（例如本地 Ollama） |
| `implicit auth` | 凭证是隐式的（例如 Vertex AI Application Default Credentials） |
| `custom auth` | `class_path` 提供商管理自己的认证（mTLS、JWT、自定义 headers） |

在某行上按 Enter 粘贴密钥，或使用行的删除功能清除现有密钥。保存的密钥以原子方式写入，跨会话和 `/reload` 持久化。

> 密钥限定到你在此机器上的用户账户——Deep Agents Code 除了发送到配置的提供商 API 外，不会将它们传输到任何地方。

`/auth` 只管理 **LLM 提供商**凭证。工具凭证如 `TAVILY_API_KEY`（网络搜索）和 `LANGSMITH_API_KEY`（追踪）从环境中读取——[在 `~/.deepagents/.env` 或 shell 中设置](#environment-variables)。

### 环境变量（CI 和无头模式）

对于非交互式运行、CI/CD 管道或没有 TUI 的地方，导出提供商的环境变量或将其写入 `~/.deepagents/.env`：

```bash
export ANTHROPIC_API_KEY="sk-ant-..."
export OPENAI_API_KEY="sk-..."
```

有关加载顺序、`DEEPAGENTS_CODE_` 前缀和每项目覆盖，请参阅下面的[环境变量](#environment-variables)。

## 环境变量

Deep Agents Code 从 dotenv 文件加载环境变量，这样你不需要在 shell 配置文件中 `export` API 密钥或在项目间重复 `.env` 文件。

### 加载顺序和优先级

启动时加载两个 `.env` 文件：

1. **项目 `.env`** — 当前工作目录中的 `.env` 文件（如果存在）
2. **全局 `~/.deepagents/.env`** — 作为所有项目回退的单个共享文件

有效优先级为：**shell 环境 > 项目 `.env` > 全局 `.env`**。shell 中已设置的值永远不会被覆盖——包括在 `/reload` 时。

### `DEEPAGENTS_CODE_` 前缀

所有 Deep Agents Code 特定的环境变量使用 `DEEPAGENTS_CODE_` 前缀（例如 `DEEPAGENTS_CODE_AUTO_UPDATE`、`DEEPAGENTS_CODE_DEBUG`）。完整列表请参阅[环境变量参考](#environment-variable-reference)。

该前缀也作为 Deep Agents Code 读取的任何环境变量的覆盖机制，包括第三方凭证。Deep Agents Code 先检查 `DEEPAGENTS_CODE_{NAME}`，然后回退到 `{NAME}`：

```bash
# 仅在 Deep Agents Code 中覆盖 OPENAI_API_KEY，不影响其他工具
DEEPAGENTS_CODE_OPENAI_API_KEY=sk-cli-only

# 通过将前缀变量设为空来阻止 Deep Agents Code 中的 shell 导出密钥
DEEPAGENTS_CODE_ANTHROPIC_API_KEY=
```

在 `/reload` 时，Deep Agents Code 重新读取 `.env` 文件并获取前缀值，因此你可以在不重启的情况下轮换密钥。

### 示例

在 `~/.deepagents/.env` 中一次性存储 API 密钥：

```bash
ANTHROPIC_API_KEY=sk-ant-...
OPENAI_API_KEY=sk-...
LANGSMITH_API_KEY=lsv2_...
TAVILY_API_KEY=tvly-...

# 如果 OPENAI_API_KEY 已在 shell 中为其他工具导出，
# 使用前缀给 Deep Agents Code 自己的密钥以避免冲突
DEEPAGENTS_CODE_OPENAI_API_KEY=sk-cli-only-...
```

然后在需要时通过在项目目录放置 `.env` 文件进行每项目覆盖。

### 使用 Tavily 启用网络搜索

内置的 `web_search` 工具使用 [Tavily](https://tavily.com)。Deep Agents Code 在启动时显示 "Web search disabled — `TAVILY_API_KEY` is not set" 通知，直到你提供密钥。与模型提供商密钥不同，Tavily **不**由 `/auth` 管理；它直接从环境读取。

1. **获取密钥：** 在 [tavily.com](https://tavily.com) 注册并复制密钥（以 `tvly-` 开头）。免费层级足以满足大多数 Deep Agents Code 使用。

2. **添加到环境：** 将密钥添加到 `~/.deepagents/.env` 以便每个会话获取：

   ```bash
   TAVILY_API_KEY=tvly-...
   ```

   或在 shell 中导出用于一次性会话：

   ```bash
   export TAVILY_API_KEY=tvly-...
   ```

   Shell 导出优先于 `.env` 值。要将密钥限定到 Deep Agents Code 而不影响其他工具，使用 [`DEEPAGENTS_CODE_` 前缀](#deepagents_code_-prefix)：`DEEPAGENTS_CODE_TAVILY_API_KEY=tvly-...`。

3. **重新加载或重启：** 在现有会话中，运行 `/reload` 重新读取 `.env` 文件。下次启动时，"Web search disabled" 通知消失，agent 可以调用 `web_search`。

## 配置文件

`~/.deepagents/config.toml` 让你自定义模型提供商、设置默认值并向模型构造函数传递额外参数。

### 默认和最近模型

```toml
[models]
default = "ollama:qwen3:4b"             # 你的长期偏好
recent = "google_genai:gemini-3.5-flash"   # 最近的 /model 切换（自动写入）
```

`[models].default` 始终优先于 `[models].recent`。`/model` 命令只写入 `[models].recent`，因此你配置的默认值不会被会话中的切换覆盖。要移除默认值，使用 `/model --default --clear` 或从配置文件中删除 `default` 键。

### 默认和最近 agent

```toml
[agents]
default = "backend-dev"  # 你的长期偏好（在 /agents 选择器中 Ctrl+S）
recent = "frontend-dev"  # 最近的 /agents 切换（自动写入）
```

`[agents].default` 始终优先于 `[agents].recent`。在 `/agents` 选择器中按 `Enter` 选择 agent 会写入 `recent`；在高亮行上按 `Ctrl+S` 将其固定为 `default`。再次在同一行上按 `Ctrl+S` 清除默认值。

显式的 `-a`/`--agent` 始终覆盖两者，`-r`/`--resume` 绕过两者以恢复线程的原始 agent。

### 提供商配置

每个提供商是 `[models.providers]` 下的一个 TOML 表：

```toml
[models.providers.<name>]
models = ["gpt-4o"]
api_key_env = "OPENAI_API_KEY"
base_url = "https://api.openai.com/v1"
class_path = "my_package.models:MyChatModel"
enabled = true

[models.providers.<name>.params]
temperature = 0
max_tokens = 4096

[models.providers.<name>.params."gpt-4o"]
temperature = 0.7
```

提供商有以下配置选项：

- **`models`**（可选，字符串列表）— 在交互式 `/model` 切换器中显示的模型名称列表。对于已经附带模型配置文件的提供商，你在此添加的任何名称会与捆绑的一起出现——对尚未添加到包中的新发布模型很有用。对于[任意提供商](#arbitrary-providers)，此列表是切换器中模型的唯一来源。此处列出的模型**绕过**基于配置文件的过滤标准。此键是可选的。你始终可以直接向 `/model` 或 `--model` 传递任何模型名称。

- **`api_key_env`**（可选，字符串）— 持有 API 密钥的环境变量的**名称**（例如 `"OPENAI_API_KEY"`），而不是密钥本身。

- **`base_url`**（可选，字符串）— 覆盖提供商使用的 base URL。

- **`params`**（可选，对象）— 转发给模型构造函数的额外关键字参数。扁平键适用于该提供商的每个模型。模型键控的子表仅覆盖该模型的单个值；合并是浅合并（模型在冲突时胜出）。不要在 `params` 中放置凭证。

- **`profile`**（可选，对象）— 覆盖模型运行时配置文件中的字段（例如 `max_input_tokens`）。扁平键适用于该提供商的每个模型。模型键控的子表仅覆盖该模型的单个值；合并是浅合并。

- **`class_path`**（可选，字符串）— 用于[任意模型](#arbitrary-providers)提供商。`module.path:ClassName` 格式的完全限定 Python 类。设置后，Deep Agents Code 直接导入并实例化此类。

- **`enabled`**（可选，布尔值，默认 `true`）— 此提供商是否出现在 `/model` 选择器中。设为 `false` 可隐藏从已安装包自动发现的提供商。你仍然可以通过 `/model provider:model` 或 `--model` 直接使用禁用的提供商。

### 模型构造函数参数

任何提供商都可以使用 `params` 表向模型构造函数传递额外参数：

```toml
[models.providers.ollama.params]
temperature = 0
num_ctx = 8192
```

#### 每模型覆盖

如果特定模型需要不同的参数，在 `params` 下添加模型键控的子表来覆盖单个值，而无需重复整个提供商配置：

```toml
[models.providers.ollama]
models = ["qwen3:4b", "llama3"]

[models.providers.ollama.params]
temperature = 0
num_ctx = 8192

[models.providers.ollama.params."qwen3:4b"]
temperature = 0.5
num_ctx = 4000
```

使用此配置：
- `ollama:qwen3:4b` 得到 `{temperature: 0.5, num_ctx: 4000}` — 模型覆盖胜出。
- `ollama:llama3` 得到 `{temperature: 0, num_ctx: 8192}` — 无覆盖，仅提供商级别参数。

合并是浅合并：模型子表中存在的任何键替换提供商级别参数中的相同键，而仅在提供商级别的键被保留。

> **提示：** 要进行一次性调整而不编辑 `config.toml`，在启动时通过 `--model-params` 传递 JSON 对象或在会话中使用 `/model`。CLI 标志优先级最高。

### 配置文件覆盖（高级）

覆盖模型运行时配置文件中的字段，改变 Deep Agents Code 解释模型能力的方式。最常见的用例是降低 `max_input_tokens` 以更早触发自动摘要——用于测试或限制上下文使用：

```toml
# 应用于该提供商的所有模型
[models.providers.anthropic.profile]
max_input_tokens = 4096
```

每模型子表与 `params` 工作方式相同——模型级别的值在冲突时胜出：

```toml
[models.providers.anthropic.profile]
max_input_tokens = 4096

# 此模型获得更高的限制
[models.providers.anthropic.profile."claude-sonnet-4-5"]
max_input_tokens = 8192
```

配置文件覆盖在模型创建后合并到模型的配置文件中。任何读取配置文件的功能——状态栏中的上下文限制显示、自动摘要阈值、能力检查——都会看到覆盖的值。

#### CLI 配置文件覆盖 `--profile-override`（高级）

要在运行时覆盖模型配置文件字段而不编辑配置文件，通过 `--profile-override` 传递 JSON 对象：

```bash
dcode --profile-override '{"max_input_tokens": 4096}'

# 与 --model 组合
dcode --model google_genai:gemini-3.5-flash --profile-override '{"max_input_tokens": 4096}'

# 在非交互模式中
dcode -n "Summarize this repo" --profile-override '{"max_input_tokens": 4096}'
```

这些值在配置文件配置文件覆盖之上合并（CLI 胜出）。优先级链为：模型默认值 < config.toml 配置文件 < CLI `--profile-override`。

`--profile-override` 值跨会话中 `/model` 热切换持久——切换模型会将覆盖重新应用到新模型。

### 自定义 base URL

一些提供商包接受 `base_url` 来覆盖默认端点。例如，`langchain-ollama` 默认通过底层 `ollama` 客户端使用 `http://localhost:11434`。要指向其他位置，在配置中设置 `base_url`：

```toml
[models.providers.ollama]
base_url = "http://your-host-here:port"
```

### 兼容 API

对于暴露与 OpenAI 或 Anthropic 兼容的 API 的提供商，你可以通过将 `base_url` 指向提供商的端点来使用现有的 `langchain-openai` 或 `langchain-anthropic` 包：

```toml
[models.providers.openai]
base_url = "https://api.example.com/v1"
api_key_env = "EXAMPLE_API_KEY"
models = ["my-model"]
```

```toml
[models.providers.anthropic]
base_url = "https://api.example.com"
api_key_env = "EXAMPLE_API_KEY"
models = ["my-model"]
```

> 提供商在官方规范之上添加的任何功能将不会被捕获。如果提供商提供专用的 LangChain 集成包，请优先使用。

> **警告：** OpenAI 提供商默认使用 [Responses API](https://platform.openai.com/docs/api-reference/responses)，大多数 OpenAI 兼容网关不实现此 API。如果你的提供商仅支持 Chat Completions API，调用可能会失败。请显式禁用 Responses API：
> ```toml
> [models.providers.openai.params]
> use_responses_api = false
> ```

### 将模型添加到交互式切换器

一些提供商（例如 `langchain-ollama`）不附带模型配置文件数据。在这种情况下，交互式 `/model` 切换器不会列出该提供商的模型。你可以通过在配置文件中为提供商定义 `models` 列表来填补空白：

```toml
[models.providers.ollama]
models = ["llama3", "mistral", "codellama"]
```

`/model` 切换器现在将包含一个 Ollama 部分，列出这些模型。

这完全是可选的。你始终可以通过直接指定完整名称切换到任何模型：

```
/model ollama:llama3
```

> 当安装了 `langchain-ollama` 且守护进程可达时，Deep Agents Code 自动发现本地拉取的模型并将它们合并到切换器中——不需要 `models` 列表。运行 `/reload` 以在拉取新模型后刷新，或设置 `DEEPAGENTS_CODE_OLLAMA_DISCOVERY=0` 退出。

### 任意提供商

你可以使用 `class_path` 使用任何 [LangChain `BaseChatModel`](https://reference.langchain.com/python/langchain_core/language_models/#langchain_core.language_models.BaseChatModel) 子类。Deep Agents Code 直接导入并实例化该类——不需要内置提供商包。

```toml
[models.providers.my_custom]
class_path = "my_package.models:MyChatModel"
api_key_env = "MY_API_KEY"
base_url = "https://my-endpoint.example.com"

[models.providers.my_custom.params]
temperature = 0
max_tokens = 4096
```

`api_key_env` 和 `base_url` 是可选的。`class_path` 提供商应该在内部处理自己的认证——当你的模型使用自定义认证（JWT 令牌、专有 headers、mTLS 等）而不是标准 API 密钥时很有用：

```toml
[models.providers.xyz]
class_path = "abc.integrations.deepagents:DeepAgentsXYZChat"
models = ["abc-xyz-1"]

[models.providers.xyz.params]
bypass_auth = true
temperature = 0
```

使用此配置，通过 `/model xyz:abc-xyz-1` 或 `--model xyz:abc-xyz-1` 切换到该模型。

> **注意：** Deep Agents Code 需要**工具调用**支持。如果你的自定义模型支持工具调用但 Deep Agents Code 不知道，请在提供商配置文件中声明：
> ```toml
> [models.providers.xyz.profile]
> tool_calling = true
> max_input_tokens = 128000
> ```

提供商包必须安装在与 `deepagents-code` 相同的 Python 环境中：

```bash
# 如果 deepagents-code 是用 uv tool 安装的：
uv tool install deepagents-code --with my_package
```

当你切换到 `my_custom:my-model-v1` 时，模型名称作为 `model` kwargs 传递：

```python
MyChatModel(model="my-model-v1", base_url="...", api_key="...", temperature=0, max_tokens=4096)
```

> **警告：** `class_path` 从你的配置文件执行任意 Python 代码。这与 `pyproject.toml` 构建脚本具有相同的信任模型——你控制自己的机器。

## Skills 额外允许的目录

默认情况下，当 Deep Agents Code 加载 skills 时，它验证解析的 skill 文件路径是否保持在标准 skill 目录内。这可以防止 skill 目录中的符号链接读取这些根目录之外的任意文件。

如果你将共享 skill 资产存储在非标准位置并使用从标准 skill 目录到这些位置的符号链接，你可以将该位置添加到包含允许列表中。这**不会**添加新的 skill 发现位置：skills 仍然只从标准目录发现。

在 `config.toml` 中配置：

```toml
[skills]
extra_allowed_dirs = [
    "~/shared-skills",
    "/opt/team-skills",
]
```

或者，设置 `DEEPAGENTS_CODE_EXTRA_SKILLS_DIRS` 环境变量作为冒号分隔的列表：

```bash
export DEEPAGENTS_CODE_EXTRA_SKILLS_DIRS="~/shared-skills:/opt/team-skills"
```

当设置环境变量时，它优先于配置文件值。更改在 `/reload` 时生效。

## 主题

使用 `/theme` 打开交互式主题选择器。浏览列表实时预览主题，按 `Enter` 将你的选择持久化到 `config.toml`。

Deep Agents Code 附带许多内置主题。默认主题是 `langchain`，一个带有 LangChain 品牌颜色的深色主题。选择的主题持久化在 `[ui]` 下：

```toml
[ui]
theme = "langchain-dark"
```

### 用户定义主题

在 `config.toml` 的 `[themes.<name>]` 部分下定义自定义主题。每个部分需要 `label`（字符串）。`dark`（布尔值）如果省略默认为 `false`——对于深色主题设为 `true`。所有颜色字段都是可选的——省略的字段根据 `dark` 标志回退到内置的深色或浅色调色板。

```toml
[themes.my-solarized]
label = "My Solarized"
dark = true
primary = "#268BD2"
warning = "#B58900"

# 带空格的主题名称需要 TOML 引用
[themes."ocean breeze"]
label = "Ocean Breeze"
primary = "#0077B6"
background = "#CAF0F8"
```

用户定义主题在 `/theme` 选择器中与内置主题一起显示。

### 覆盖内置主题颜色

要调整内置主题的颜色而不创建新主题，使用 `[themes.<builtin-name>]` 部分。只读取颜色字段——`label` 和 `dark` 从内置主题继承：

```toml
[themes.langchain]
primary = "#FF5500"
```

省略的颜色字段保留现有的内置值。

`[themes.*]` 部分的更改在 `/reload` 时生效。

### 将主题映射到终端

如果你在具有不同配色方案的终端之间切换（例如深色 iTerm 和浅色 Apple Terminal），在 `[ui.terminal_themes]` 下将每个终端映射到一个主题。Deep Agents Code 匹配 shell 的 `TERM_PROGRAM` 并自动应用映射的主题：

```toml
[ui.terminal_themes]
"Apple_Terminal" = "langchain-light"
"iTerm.app" = "langchain"
```

在 `/theme` 选择器中按 `T` 为当前终端保存高亮主题，或运行 `echo $TERM_PROGRAM` 查找你的终端标识符并手动添加。

**主题解析顺序：**

1. `DEEPAGENTS_CODE_THEME` 环境变量（显式覆盖）。
2. 当前 `TERM_PROGRAM` 的 `[ui.terminal_themes]` 映射。
3. `[ui] theme` 保存的偏好（由 `/theme` 设置）。
4. 内置默认值（`langchain`）。

## 自动更新

Deep Agents Code 可以自动检查和安装更新。

在 `config.toml` 中配置：

```toml
[update]
auto_update = true
```

或通过环境变量：

```bash
export DEEPAGENTS_CODE_AUTO_UPDATE=1
```

环境变量优先于配置文件。

启用后，Deep Agents Code 在会话开始时检查 PyPI 是否有更新版本，并使用检测到的安装方法（uv、Homebrew 或 pip）自动升级。禁用时（默认），Deep Agents Code 显示带有适当安装命令的更新提示。

你也可以随时使用 `/update` 斜杠命令手动检查和安装更新，它绕过缓存并内联报告成功或失败。

升级后，Deep Agents Code 在下次启动时显示 "what's new" 横幅，带有 changelog 链接。

## 托管部署

安装脚本支持以 root 身份运行，目标是 macOS MDM 工具（Kandji、Jamf 等），这些工具在最小 root 环境中执行脚本。

当 `id -u` 为 `0` 时，脚本：

1. 解析真实控制台用户的 `HOME`（通过 `/dev/console` 或 `/Users` 目录扫描）
2. 在每个安装步骤后将所有创建的文件 `chown` 回目标用户

非 root 安装不受影响。

要为托管安装预配置自动更新，在用户的 shell 配置文件中设置 `DEEPAGENTS_CODE_AUTO_UPDATE=1` 或部署带 `[update] auto_update = true` 的 `config.toml` 到 `~/.deepagents/config.toml`。要完全抑制自动更新和更新检查，设置 `DEEPAGENTS_CODE_NO_UPDATE_CHECK=1`。

## Hooks

Hooks 让外部程序对 Deep Agents Code 生命周期事件做出反应。在 `~/.deepagents/hooks.json` 中配置命令，每当事件触发时，它将 JSON 载荷通过管道传输到每个匹配命令的 stdin。

Hooks 以 fire-and-forget 方式在后台线程中运行——它们永远不会阻塞 Deep Agents Code，失败被记录而不会中断你的会话。

### 设置

创建 `~/.deepagents/hooks.json`：

```json
{
  "hooks": [
    {
      "command": ["bash", "-c", "cat >> ~/deepagents-events.log"],
      "events": ["session.start", "session.end"]
    }
  ]
}
```

现在每次会话开始或结束时，Deep Agents Code 将事件载荷追加到 `~/deepagents-events.log`。

### Hook 配置

配置文件包含单个 `hooks` 数组。每个条目有：

- **`command`**（必需，字符串列表）— 要运行的命令和参数。无 shell 展开：如需要使用 `["bash", "-c", "..."]`。
- **`events`**（可选，字符串列表）— 要订阅的事件名称。省略或留空以接收**所有**事件。

```json
{
  "hooks": [
    {
      "command": ["python3", "my_handler.py"],
      "events": ["session.start", "task.complete"]
    },
    {
      "command": ["bash", "log_everything.sh"]
    }
  ]
}
```

第二个 hook 没有 `events` 过滤器，因此它接收 Deep Agents Code 发出的每个事件。

### 载荷格式

每个 hook 命令在 stdin 上接收一个 JSON 对象，带有 `"event"` 键加上事件特定字段：

```json
{
  "event": "session.start",
  "thread_id": "abc123"
}
```

### 事件参考

| 事件 | 何时触发 | 附加字段 |
|------|---------|---------|
| `session.start` | agent 会话开始时 | `thread_id` |
| `session.end` | 会话退出时 | `thread_id` |
| `user.prompt` | 用户在交互模式中提交聊天消息时 | 无 |
| `input.required` | agent 需要人工输入时（human-in-the-loop 中断） | 无 |
| `permission.request` | 审批对话框之前，当一个或多个工具调用需要用户许可时 | `tool_names` |
| `tool.error` | 工具调用返回错误时 | `tool_names` |
| `task.complete` | agent 完成当前任务时 | `thread_id` |
| `context.compact` | Deep Agents Code 压缩（摘要）对话上下文之前 | 无 |

### 执行模型

- **后台线程**：Hook 子进程通过 `asyncio.to_thread` 在线程中运行，因此主事件循环永远不会被阻塞。
- **并发分发**：当多个 hooks 匹配一个事件时，它们在线程池中并发运行。
- **5 秒超时**：每个命令有 5 秒超时。超过此时间的命令被杀死。
- **Fire-and-forget**：错误被每个 hook 捕获并在 debug/warning 级别记录。失败的 hook 永远不会崩溃或阻塞 Deep Agents Code。
- **延迟加载**：配置文件在第一次事件分发时读取一次，并在会话其余时间缓存。
- **无 shell 展开**：命令直接执行（不通过 shell）。如需要 shell 功能如管道或变量展开，用 `["bash", "-c", "..."]` 包装。

### Hook 示例

**记录所有事件到文件：**

```json
{
  "hooks": [
    {
      "command": ["bash", "-c", "jq -c . >> ~/.deepagents/hook-events.jsonl"],
      "events": []
    }
  ]
}
```

**任务完成时桌面通知（macOS）：**

```json
{
  "hooks": [
    {
      "command": [
        "bash", "-c",
        "osascript -e 'display notification \"Agent finished\" with title \"Deep Agents\"'"
      ],
      "events": ["task.complete"]
    }
  ]
}
```

**Python 处理器：**

```python
# my_handler.py
import json
import sys

payload = json.load(sys.stdin)
event = payload["event"]

if event == "session.start":
    print(f"Session started: {payload['thread_id']}", file=sys.stderr)
elif event == "permission.request":
    print(f"Approval needed for: {payload['tool_names']}", file=sys.stderr)
```

```json
{
  "hooks": [
    {
      "command": ["python3", "my_handler.py"],
      "events": ["session.start", "permission.request"]
    }
  ]
}
```

### 安全注意事项

Hooks 遵循与 Git hooks 或 shell 别名相同的信任模型——任何可以写入 `~/.deepagents/hooks.json` 的用户都可以执行任意命令。这是设计使然：

- **无命令注入**：载荷数据仅作为 JSON 流向 stdin，永远不会到命令行参数。`json.dumps` 处理转义。
- **默认无 shell**：命令以 `shell=False` 运行，防止 shell 注入。
- **格式错误的配置**：无效的 JSON 或意外类型产生记录的警告，而不是安全问题。

> **警告：** 只添加你信任来源的 hooks。hook 与你的用户账户具有相同的权限。
