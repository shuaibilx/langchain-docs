# MCP Tools

> 从 MCP（Model Context Protocol）服务器加载额外工具

[MCP（Model Context Protocol）](https://modelcontextprotocol.io/) 让你可以用来自外部服务器的工具扩展 Deep Agents Code——文件系统、API、数据库等——无需修改 agent 本身。Deep Agents Code 在启动时连接到 MCP 服务器，发现它们的工具，并使它们与内置工具一起可供 agent 使用。

通过将 `.mcp.json` 配置文件添加到项目中来添加 MCP 服务器（项目级作用域），或在用户级别添加以应用于所有项目。

## 快速开始

本快速入门将 [LangChain 文档 MCP 服务器](https://docs.langchain.com/mcp) 添加到你机器上的每个 Deep Agents Code 会话。可以用任何其他 MCP 服务器的 URL 或 stdio 命令替换。

### 步骤 1：创建配置文件

如果尚不存在，在用户级别创建 `.mcp.json` 文件，使服务器对机器上的每个项目可用，或在项目级别创建。

**用户级别：**

```bash
mkdir -p ~/.deepagents
touch ~/.deepagents/.mcp.json
```

此文件（`~/.deepagents/.mcp.json`）中的服务器在此机器上的每个项目中可用。

**项目级别：**

```bash
touch .mcp.json
```

此文件（`<project>/.mcp.json`）中的服务器仅对此项目可用。

**项目级别（隐藏）：**

```bash
mkdir -p .deepagents
touch .deepagents/.mcp.json
```

此文件（`<project>/.deepagents/.mcp.json`）中的服务器仅对此项目可用，但不放在仓库根目录。

详见[发现位置](#discovery-locations)了解完整优先级规则。

### 步骤 2：添加 MCP 服务器

```json
{
    "mcpServers": {
        "docs-langchain": {
            "type": "http",
            "url": "https://docs.langchain.com/mcp"
        }
    }
}
```

要添加更多服务器，在 `mcpServers` 中添加更多条目。详见[配置格式](#configuration-format)了解 OAuth、stdio、SSE 和 HTTP 服务器字段、环境变量和 headers。

### 步骤 3：启动 Deep Agents Code

```bash
dcode
```

启动时，Deep Agents Code 自动发现配置，连接到每个服务器，发现其工具，并打印确认：

```
✓ Loaded 3 MCP tools
```

在交互式会话中运行 `/mcp` 可以查看每个服务器的状态、传输方式和已加载的工具列表。agent 现在可以在会话期间使用这些工具——stdio 服务器在工具调用之间保持活动状态。

## 自动发现

Deep Agents Code 自动在标准位置搜索 `.mcp.json` 文件。无需标志——只需放置配置文件就会被自动发现。

### 发现位置

配置按以下顺序检查（从低到高优先级）：

| 优先级 | 位置 | 作用域 |
|--------|------|--------|
| 1（最低） | `~/.deepagents/.mcp.json` | 用户级——适用于所有项目 |
| 2 | `<project>/.deepagents/.mcp.json` | 项目级——`.deepagents` 子目录 |
| 3（最高） | `<project>/.mcp.json` | 项目级——根目录（Claude Code 兼容） |

项目根目录是包含 `.git` 文件夹的最近父目录，回退到当前工作目录。

当存在多个配置文件时，它们的 `mcpServers` 条目会被合并。如果同一服务器名称出现在多个文件中，更高优先级的配置胜出。这允许项目级配置覆盖用户级条目（例如，固定同一服务器的不同版本）而不影响其他项目。

### 标志

| 标志 | 行为 |
|------|------|
| `--mcp-config PATH` | 添加显式配置作为最高优先级来源（在自动发现的配置之上合并） |
| `--no-mcp` | 完全禁用 MCP——不加载任何服务器 |

> `--mcp-config` 和 `--no-mcp` 互斥。

### Claude Code 兼容性

如果你已经在项目根目录有为 Claude Code 准备的 `.mcp.json`，Deep Agents Code 会自动获取它——无需额外设置。

## 配置格式

`mcpServers` 下的每个键是服务器名称。服务器的字段决定 Deep Agents Code 如何连接到它。

### stdio 服务器（默认）

stdio 服务器作为子进程生成。Deep Agents Code 通过 stdin/stdout 与它们通信。

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/tmp"],
      "env": {}
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "your-token" }
    }
  }
}
```

### SSE 和 HTTP 服务器

对于远程 MCP 服务器，将 `type` 设为 `"sse"` 或 `"http"` 并提供 `url`：

```json
{
  "mcpServers": {
    "remote-api": {
      "type": "sse",
      "url": "https://api.example.com/mcp",
      "headers": { "Authorization": "Bearer your-token" }
    }
  }
}
```

### 字段参考

**stdio（默认）：** 必需：`command`。可选：`args`、`env`，加上共享的[工具过滤字段](#tool-filtering)。

- `command`（必需，字符串）— 要运行的可执行文件。
- `args`（可选，字符串列表）— 传递给命令的参数。
- `env`（可选，对象）— 为子进程设置的环境变量。用于传递 API 密钥和其他凭证而不暴露在 shell 历史中。

**sse：** 必需：`type: "sse"`、`url`。可选：`headers`、`auth`，加上共享的[工具过滤字段](#tool-filtering)。

- `type`（必需，`"sse"`）— 传输类型。
- `url`（必需，字符串）— 服务器端点 URL。
- `headers`（可选，对象）— 每个请求发送的 HTTP headers。通常用于认证。值支持 `${VAR}` 引用父 shell 环境变量（在服务器激活时解析）。
- `auth`（可选，`"oauth"`）— 设为 `"oauth"` 以通过 `dcode mcp login` 驱动 OAuth 登录流程，而不是提供 `Authorization` header。不能与 `Authorization` header 组合。详见 [OAuth 登录](#oauth-login)。

**http：** 必需：`type: "http"`、`url`。可选：`headers`、`auth`，加上共享的[工具过滤字段](#tool-filtering)。

- `type`（必需，`"http"`）— 传输类型。使用 `"http"` 进行可流式传输的 HTTP。`streamable_http` 和 `streamable-http` 作为别名也被接受。
- `url`（必需，字符串）— 服务器端点 URL。
- `headers`（可选，对象）— 每个请求发送的 HTTP headers。值支持 `${VAR}` 引用父 shell 环境变量。
- `auth`（可选，`"oauth"`）— 设为 `"oauth"` 以驱动 OAuth 登录流程。

> `type` 字段也可以写成 `transport` 以与其他 MCP 客户端兼容。

> 服务器名称必须匹配 `[A-Za-z0-9_-]+`。名称用作 OAuth 令牌文件的磁盘基本名称，因此路径分隔符和其他 shell 元字符在配置加载时被拒绝。

### Header 环境变量

Header 值支持来自父 shell 的 `${VAR}` 替换，在服务器激活时解析而不是在配置加载时。一个未设置的变量只会使需要它的服务器失败；其他服务器仍然正常启动。

```json
{
    "mcpServers": {
        "internal-api": {
            "type": "http",
            "url": "https://api.example.com/mcp",
            "headers": { "Authorization": "Bearer ${INTERNAL_API_TOKEN}" }
        }
    }
}
```

## 多个服务器

你可以根据需要配置任意多的服务器。所有服务器的工具被合并并可供 agent 使用：

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/home/user/projects"]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "ghp_..." }
    },
    "database": {
      "type": "sse",
      "url": "https://db-mcp.internal:8080/mcp",
      "headers": { "Authorization": "Bearer ..." }
    }
  }
}
```

## 工具过滤

每个服务器可以通过两个可选字段之一缩小它向 agent 暴露的工具：

- `allowedTools`：仅保留列出的工具；丢弃其他所有。
- `disabledTools`：丢弃列出的工具；保留其他所有。

过滤适用于 stdio、HTTP 和 SSE 服务器。以下两种情况在配置加载时被拒绝：

- 在同一服务器上同时设置 `allowedTools` 和 `disabledTools`。
- 将任一字段设为空列表（会静默剥离所有工具，或为空操作）。省略该字段即可。

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/tmp"],
      "allowedTools": ["read_file", "list_directory"]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "disabledTools": ["delete_repository", "delete_*_branch"]
    }
  }
}
```

### 匹配规则

每个条目是字面工具名称或 [`fnmatch`](https://docs.python.org/3/library/fnmatch.html) 风格的 glob 模式（任何包含 `*`、`?` 或 `[` 的条目被视为模式）。条目同时匹配裸 MCP 工具名称和服务器前缀形式（`{server}_{tool}`），因此两种形式都可以使用：

```json
{
  "allowedTools": ["read_file", "fs_list_*"]
}
```

> 不匹配任何已加载工具的条目被记录为警告而不是错误——底层 MCP 服务器可以在版本间演变其工具列表而不会破坏你的配置。

- `allowedTools`（字符串列表）— 要保留的工具名称或 `fnmatch` glob 模式。此服务器的所有其他工具被丢弃。与 `disabledTools` 互斥。
- `disabledTools`（字符串列表）— 要丢弃的工具名称或 `fnmatch` glob 模式。此服务器的所有其他工具被保留。与 `allowedTools` 互斥。

## OAuth 登录

对于需要 OAuth 的远程 MCP 服务器（Slack、GitHub、Notion、Linear 和其他托管 MCP 端点），在服务器条目上设置 `"auth": "oauth"` 并运行一次登录子命令。令牌被持久化到磁盘并自动刷新。

### 配置服务器

```json
{
    "mcpServers": {
        "linear": {
            "type": "http",
            "url": "https://mcp.linear.app/mcp",
            "auth": "oauth"
        }
    }
}
```

`auth: "oauth"` 与同一条目上的 `Authorization` header 互斥，不能在 stdio 服务器上设置。

### 运行登录流程

```bash
dcode mcp login linear
```

发生什么取决于服务器的主机：

- **符合规范的服务器**（默认）：Deep Agents Code 执行动态客户端注册，在浏览器中打开授权码 + PKCE 流程，并要求你将重定向的 URL 粘贴回终端。
- **Slack**（`slack.com`、`*.slack.com`）：相同的粘贴回流程，但预置了 Slack 的公共客户端。你会被提示输入可选的团队 ID（例如 `T01234567`），以便应用安装到正确的工作区。
- **GitHub**（`api.githubcopilot.com`）：RFC 8628 设备授权授权。Deep Agents Code 打印验证 URL 和用户代码；你在浏览器中输入代码，Deep Agents Code 轮询完成。

默认情况下，`dcode mcp login` 读取与 Deep Agents Code 在运行时使用的相同自动发现配置（受项目级信任门控约束）。传递 `--config <path>` 使用特定文件：

```bash
dcode mcp login linear --config ./mcp-config.json
```

> **警告：** 未被信任的项目级配置（参见[项目级信任](#project-level-trust)）在 `mcp login` 期间被跳过，以防止攻击者控制的 `headers` 条目通过 `${VAR}` 插值泄露本地 secrets。在项目中运行一次 `dcode` 以批准配置，或显式传递 `--config <path>`。

### 令牌存储

令牌写入：

```
~/.deepagents/.state/mcp-tokens/<server>-<sha256-16(url)>.json
```

`<sha256-16(url)>` 段是服务器 URL 的 SHA-256 的前 16 个十六进制字符。目录锁定为 `0700` 模式，每个令牌文件为 `0600` 模式。文件包括 OAuth 访问令牌、刷新令牌和动态注册的客户端信息，全部在模式版本化的载荷中原子写入（写入临时文件 + `rename`）。

> 将 URL 哈希到文件名中意味着指向不同 URL 的同名服务器（例如 dev vs. prod）获得独立的令牌文件，不会相互覆盖。

### 重新认证

当运行时刷新失败（刷新令牌过期或被撤销）时，Deep Agents Code 将服务器标记为 `unauthenticated` 而不是使 agent 崩溃。欢迎横幅显示未认证服务器的数量，`/mcp` 报告每个服务器的原因。重新运行 `dcode mcp login <server>` 以刷新凭证——你的对话继续而不需重启。

## 服务器状态

每个配置的服务器在启动后处于三种状态之一：

| 状态 | 含义 |
|------|------|
| `ok` | 已连接；工具已加载并可供 agent 使用 |
| `unauthenticated` | 需要 OAuth 登录或刷新失败——运行 `dcode mcp login <server>` |
| `error` | 预检、发现或传输设置失败；附加了错误消息 |

单个失败的服务器不再中止启动。agent 使用成功连接的服务器运行，欢迎横幅在工具计数旁显示未认证和出错服务器的计数。在交互式会话中打开 `/mcp` 可以查看每个服务器的状态、传输方式、工具列表和非 `ok` 条目的失败原因。查看器在服务器连接时实时更新，支持 `tab`/`shift+tab` 导航。

## 项目级信任

项目级配置可能包含执行本地命令的 stdio 服务器和 `headers` 可能从你的环境插值 `${VAR}` 的远程服务器。为防止不受信任的仓库在 CLI 启动时运行任意代码或泄露本地 secrets，Deep Agents Code 对项目级条目执行**默认拒绝**策略。

### 工作原理

- **交互模式：** Deep Agents Code 在激活项目服务器前提示批准，显示每个 stdio 命令和远程 URL。批准使用 SHA-256 内容指纹持久化——如果配置更改，你会被再次提示。
- **非交互模式（`-n`）：** 除非传递 `--trust-project-mcp`，否则项目服务器被静默跳过。
- **信任覆盖 stdio 和远程条目**——远程服务器在预检探测期间可能 SSRF 到 localhost 或云元数据端点，并通过 headers 泄露 `${VAR}` 值，因此它们与 stdio 一样被门控。
- **用户级配置**（`~/.deepagents/.mcp.json`）始终被信任——与 `config.toml` 和 `hooks.json` 相同的信任模型。
- **`dcode mcp login`** 也遵守项目信任：在登录发现期间跳过不受信任的项目级配置，以防止攻击者控制的远程条目在 OAuth 握手中获取 secrets。

### 标志

| 标志 | 行为 |
|------|------|
| `--trust-project-mcp` | 不提示即信任所有项目级 stdio 服务器（用于 CI 和自动化） |

```bash
# 跳过批准提示
dcode --trust-project-mcp

# 非交互：显式信任项目服务器
dcode -n "run tests" --trust-project-mcp
```

### 信任存储

信任决策存储在 `~/.deepagents/config.toml` 中：

```toml
[mcp_trust.projects]
"/Users/you/myproject" = "sha256:abc123..."
```

每个键是绝对项目根路径。值是连接的项目级配置内容的 SHA-256 摘要。要撤销信任，删除条目或修改项目的 `.mcp.json`（会自动使指纹失效）。

> **警告：** 被信任的 stdio MCP 服务器与你的用户账户具有相同的权限。只批准你信任的仓库的服务器。在接受前审查批准提示中显示的命令。

## 系统提示感知

已连接的 MCP 服务器及其工具自动列在 agent 的系统提示中，按服务器名称和传输类型分组。这有助于模型推理工具来源和故障域，而无需手动上下文。

## 故障排除

**服务器启动失败（stdio）：** 验证命令在 Deep Agents Code 外部是否工作：

```bash
npx -y @modelcontextprotocol/server-filesystem /tmp
```

常见原因：包未安装、`npx` 不在 `PATH` 上、或缺少必需的环境变量。

**连接被拒绝（SSE/HTTP）：** 检查远程服务器是否正在运行以及 URL 是否正确。如果服务器需要认证，确保 `headers` 包含正确的凭证。

**工具未出现：** Deep Agents Code 在启动时打印加载的工具数量（例如 `✓ Loaded 3 MCP tools`）。如果你看到 `0`，服务器成功启动但未公布任何工具——检查服务器自己的日志或文档。

**服务器在 /mcp 中显示 `unauthenticated`：** 要么你还没有运行 `dcode mcp login <server>`，要么持久化的刷新令牌过期或在服务器端被撤销。重新运行登录命令——你的会话继续运行，服务器将在令牌刷新后重新附加。

**`Invalid MCP config at ...`：** 预检验证拒绝了 `--mcp-config`（或自动发现的 `.mcp.json`）。常见原因：不支持的服务器名称（必须匹配 `[A-Za-z0-9_-]+`）、stdio 服务器上的 `auth: oauth`、同一条目上同时设置了 `command` 和 `url`、或 header 值不是字符串。修复突出显示的原因并重新启动。

**`${VAR}` header 引用失败：** Header 插值在激活时运行，因此未设置的变量只会使需要它的服务器失败。在父 shell 中导出变量或将其添加到 `~/.deepagents/.env`。要调试，设置 `DEEPAGENTS_CODE_DEBUG=1` 并检查关闭时打印到 stderr 的每会话日志路径。
