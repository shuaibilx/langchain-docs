# Deep Agents 中的上下文工程

> 控制你的 deep agent 可以访问哪些上下文，以及在长时间运行的任务中如何管理上下文

上下文工程是以正确的格式提供正确的信息和工具，使你的 deep agent 能够可靠地完成任务。

Deep agents 可以访问几种上下文。一些来源在代理启动时提供；其他在运行时可用，例如用户输入。Deep agents 包含内置机制来管理长时间运行的会话中的上下文。

本页概述了你的 deep agent 可以访问和管理的不同类型上下文。

## 上下文类型

| 上下文类型 | 你控制什么 | 范围 |
|-----------|-----------|------|
| **[输入上下文](#输入上下文)** | 代理启动时提示中的内容（系统提示、memory、skills） | 静态，每次运行应用 |
| **[运行时上下文](#运行时上下文)** | 调用时传递的静态配置（用户元数据、API keys、连接） | 每次运行，传播到子代理 |
| **[上下文压缩](#上下文压缩)** | 内置的卸载和摘要，保持上下文在窗口限制内 | 自动，接近限制时触发 |
| **[上下文隔离（子代理）](#使用子代理进行上下文隔离)** | 使用子代理隔离繁重工作，仅返回结果给主代理 | 每个子代理，委派时触发 |
| **[长期记忆](#长期记忆)** | 使用虚拟文件系统跨线程持久化存储 | 跨对话持久化 |

## 输入上下文

输入上下文是在代理启动时提供的信息，成为其系统提示的一部分。最终提示由几个来源组成：

* **系统提示**：你提供的自定义指令加上内置代理指导。
* **Memory**：配置后始终加载的持久化 `AGENTS.md` 文件。
* **Skills**：按需加载的能力，仅在相关时加载（渐进式披露）。
* **工具提示**：使用内置工具或自定义工具的指令。

### 系统提示

你的自定义系统提示被前置到内置系统提示之前，内置系统提示包括规划、文件系统工具和子代理的指导。用它来定义代理的角色、行为和知识：

```python
from deepagents import create_deep_agent

agent = create_deep_agent(
    model="google_genai:gemini-3.1-pro-preview",
    system_prompt=(
        "You are a research assistant specializing in scientific literature. "
        "Always cite sources. Use subagents for parallel research on different topics."
    ),
)
```

`system_prompt` 参数是静态的，这意味着它不会每次调用都改变。对于某些用例，你可能想要动态提示：例如，告诉模型"You have admin access"vs"You have read-only access"，或从[长期记忆](#长期记忆)注入用户偏好如"User prefers concise responses"。如果你的提示依赖于上下文或 `runtime.store`，使用 `@dynamic_prompt` 构建上下文感知指令。你的 middleware 可以读取 `request.runtime.context` 和 `request.runtime.store`。

当工具单独使用上下文或 `runtime.store` 时，你**不需要** middleware；工具直接接收 [ToolRuntime](https://reference.langchain.com/python/langchain/tools/#langchain.tools.ToolRuntime) 对象（包括 `runtime.context` 和 `runtime.store`）。仅当工具需要与系统提示更新打包在一起时才添加 middleware。

> **提示：** 要为特定提供商或模型调整组装的系统提示，使用 [harness profile](/oss/python/deepagents/profiles#harness-profiles)：`base_system_prompt` 完全替换基础提示，`system_prompt_suffix` 追加到其后。

### Memory

Memory 文件（[`AGENTS.md`](https://agents.md/)）提供**始终加载**到系统提示中的持久化上下文。将 memory 用于项目约定、用户偏好和应适用于每个对话的关键指南：

```python
agent = create_deep_agent(
    model="google_genai:gemini-3.1-pro-preview",
    memory=["/project/AGENTS.md", "~/.deepagents/preferences.md"],
)
```

与 skills 不同，memory 始终注入——没有渐进式披露。保持 memory 最小化以避免上下文过载；使用 [skills](/oss/python/deepagents/skills) 获取详细工作流和领域特定内容。配置详情请参阅 [Memory](/oss/python/deepagents/customization#memory)。

### Skills

Skills 提供**按需**能力。代理在启动时读取每个 `SKILL.md` 的 frontmatter，然后仅在确定 skill 相关时才加载完整内容。这减少了 token 使用，同时仍提供专门的工作流：

```python
agent = create_deep_agent(
    model="google_genai:gemini-3.1-pro-preview",
    skills=["/skills/research/", "/skills/web-search/"],
)
```

保持每个 skill 专注于单一工作流或领域；广泛或重叠的 skills 会稀释相关性并在加载时膨胀上下文。在 skill 内，保持主要内容简洁，将详细参考资料移至 skill 文件中引用的单独文件中。将始终相关的约定放入 [memory](#memory)。编写和配置请参阅 [Skills](/oss/python/deepagents/skills)。

### 工具提示

[工具](/oss/python/langchain/tools)提示是塑造模型如何使用工具的指令。所有工具都向模型暴露元数据——通常是 schema 和描述。你通过 `tools` 参数传递的工具将该工具元数据（schema 和描述）呈现给模型。Deep agent 的内置工具打包在 middleware 中，通常还会用更多指导更新系统提示。

**内置工具** – 添加 harness 能力（规划、文件系统、子代理）的 middleware 自动向系统提示追加工具特定指令，创建解释如何有效使用这些工具的工具提示：

* 规划提示 – `write_todos` 维护结构化任务列表的指令
* 文件系统提示 – `ls`、`read_file`、`write_file`、`edit_file`、`glob`、`grep`（使用沙箱 backend 时还有 `execute`）的文档
* 子代理提示 – 使用 `task` 工具委派工作的指导
* 人机交互提示 – 在指定工具调用处暂停的用法（设置 `interrupt_on` 时）
* 本地上下文提示 – 当前目录和项目信息（仅 CLI）

**你提供的工具** – 通过 `tools` 参数传递的工具将其描述（来自工具 schema）发送给模型。你还可以添加[自定义 middleware](/oss/python/langchain/middleware) 来添加工具并追加自己的系统提示指令。

对于你提供的工具，确保提供清晰的名称、描述和参数描述。这些指导模型关于何时以及如何使用工具的推理。在描述中包含*何时*使用该工具，并描述每个参数的作用。

```python
@tool(parse_docstring=True)
def search_orders(
    user_id: str,
    status: str,
    limit: int = 10
) -> str:
    """Search for user orders by status.

    Use this when the user asks about order history or wants to check
    order status. Always filter by the provided status.

    Args:
        user_id: Unique identifier for the user
        status: Order status: 'pending', 'shipped', or 'delivered'
        limit: Maximum number of results to return
    """
    # Implementation here
    ...
```

> **提示：** 要为特定提供商或模型覆盖内置或用户提供的工具描述，使用 [harness profile](/oss/python/deepagents/profiles#harness-profiles) 的 `tool_description_overrides`，按工具名称键控。`excluded_tools` 从可见工具集中完全移除工具。

### 完整系统提示

Deep agent 的系统消息——运行开始时模型接收的组装系统提示——由以下部分组成：

1. 自定义 `system_prompt`（如果提供）
2. [基础代理提示](https://github.com/langchain-ai/deepagents/blob/e18e9dcd0e6edc72c0a4a5b76ae752c4bc539752/libs/deepagents/deepagents/graph.py#L37)
3. 待办列表提示：如何使用待办列表规划的指令
4. Memory 提示：`AGENTS.md` + memory 使用指南（仅当提供 `memory` 时）
5. Skills 提示：Skills 位置 + 带 frontmatter 信息的 skills 列表 + 用法（仅当提供 skills 时）
6. 虚拟文件系统提示（文件系统 + execute 工具文档，如适用）
7. 子代理提示：Task 工具用法
8. 用户提供的 middleware 提示（如果提供自定义 middleware）
9. 人机交互提示（设置 `interrupt_on` 时）

## 运行时上下文

运行时上下文是你调用代理时传递的每次运行配置。它不会自动包含在模型提示中；只有当工具、middleware 或其他逻辑读取它并将其添加到消息或系统提示中时，模型才能看到它。将运行时上下文用于用户元数据（ID、偏好、角色）、API keys、数据库连接、feature flags 或你的工具和 harness 需要的其他值。

使用 `context_schema` 定义数据的形状：使用 `dataclasses.dataclass` 或 `typing.TypedDict` 类。使用 **`context`** 参数传递值给 `invoke` / `ainvoke`。完整详情请参阅 [Runtime](/oss/python/langchain/runtime) 和 [LangGraph 运行时上下文](/oss/python/langgraph/graph-api#runtime-context)。

在工具内部，从注入的 [ToolRuntime](https://reference.langchain.com/python/langchain/tools/#langchain.tools.ToolRuntime) 读取上下文：

```python
from dataclasses import dataclass
from deepagents import create_deep_agent
from langchain.tools import tool, ToolRuntime

@dataclass
class Context:
    user_id: str
    api_key: str

@tool
def fetch_user_data(query: str, runtime: ToolRuntime[Context]) -> str:
    """Fetch data for the current user."""
    user_id = runtime.context.user_id
    return f"Data for user {user_id}: {query}"

agent = create_deep_agent(
    model="google_genai:gemini-3.1-pro-preview",
    tools=[fetch_user_data],
    context_schema=Context,
)

result = agent.invoke(
    {"messages": [{"role": "user", "content": "Get my recent activity"}]},
    context=Context(user_id="user-123", api_key="sk-..."),
)
```

运行时上下文**传播到所有子代理**。当子代理运行时，它接收与父代理相同的运行时上下文。每个子代理的上下文（命名空间键）请参阅 [Subagents](/oss/python/deepagents/subagents#context-management)。

## 上下文压缩

长时间运行的任务产生大量工具输出和长对话历史。上下文压缩减少代理工作记忆中信息的大小，同时保留与任务相关的细节。以下技术是确保传递给 LLM 的上下文保持在其上下文窗口限制内的内置机制：

### 卸载

Deep Agents 使用[内置文件系统工具](/oss/python/deepagents/harness#virtual-filesystem-access)自动卸载内容，并根据需要搜索和检索卸载的内容。当工具调用输入或结果超过 token 阈值（默认 20,000）时发生内容卸载：

1. **工具调用输入超过 20,000 tokens**：文件写入和编辑操作在代理的对话历史中留下包含完整文件内容的工具调用。由于此内容已持久化到文件系统，它通常是冗余的。当会话上下文超过模型可用窗口的 85% 时，deep agents 截断较旧的工具调用，用指向磁盘文件的指针替换它们，减少活动上下文的大小。

2. **工具调用结果超过 20,000 tokens**：当发生这种情况时，deep agent 将响应卸载到配置的 backend，并用文件路径引用和前 10 行预览替换它。代理可以根据需要重新读取或搜索内容。

### 摘要

当上下文大小超过模型的上下文窗口限制（例如 `max_input_tokens` 的 85%），且没有更多上下文可卸载时，deep agent 会摘要消息历史。

此过程有两个组件：

* **上下文内摘要**：LLM 生成对话的结构化摘要，包括会话意图、创建的制品和下一步——替换代理工作记忆中的完整对话历史。
* **文件系统保留**：完整的原始对话消息作为规范记录写入文件系统。

这种双重方法确保代理保持对其目标和进度的感知（通过摘要），同时保留需要时恢复特定细节的能力（通过文件系统搜索）。

**配置：**

* 在模型的 [model profile](/oss/python/langchain/models#model-profiles) 的 `max_input_tokens` 的 85% 时触发
* 保留 10% 的 tokens 作为最近上下文
* 如果 model profile 不可用，回退到 170,000-token 触发 / 保留 6 条消息
* 如果任何模型调用引发标准的 [ContextOverflowError](https://reference.langchain.com/python/langchain-core/exceptions/ContextOverflowError)，deep agent 立即回退到摘要并用摘要 + 最近保留的消息重试
* 较旧的消息由模型摘要

> **提示：** 代理的[流式 tokens](/oss/python/deepagents/streaming#llm-tokens) 通常包含摘要步骤生成的 tokens。你可以使用关联的元数据过滤掉这些 tokens：
>
> ```python
> for chunk in agent.stream(
>     {"messages": [...]},
>     stream_mode="messages",
>     version="v2",
> ):
>     token, metadata = chunk["data"]
>     if metadata.get("lc_source") == "summarization":
>         continue
>     else:
>         ...
> ```

##### 摘要工具

Deep agents 包含一个可选的[摘要工具](/oss/python/langchain/tools)，使代理能够在适当时机触发摘要——例如在任务之间——而不是在固定的 token 间隔。

你可以通过将其追加到 middleware 列表来启用此工具：

```python
from deepagents import create_deep_agent
from deepagents.backends import StateBackend
from deepagents.middleware.summarization import (
    create_summarization_tool_middleware,
)

backend = StateBackend  # if using default backend

model = "google_genai:gemini-3.1-pro-preview"
agent = create_deep_agent(
    model=model,
    middleware=[
        create_summarization_tool_middleware(model, backend),
    ],
)
```

启用此功能不会禁用在 85% 模型上下文限制时的默认摘要操作。

## 使用子代理进行上下文隔离

子代理解决了**上下文膨胀问题**。当主代理使用具有大量输出的工具（网络搜索、文件读取、数据库查询）时，上下文窗口会快速填满。子代理隔离这些工作——主代理只接收最终结果，而不是产生它的数十个工具调用。你还可以独立于主代理配置每个子代理（例如模型、工具、系统提示和 skills）。

**工作原理：**

* 主代理有一个 `task` 工具来委派工作
* 子代理使用自己的新上下文运行
* 子代理自主执行直到完成
* 子代理返回单个最终报告给主代理
* 主代理的上下文保持干净

**最佳实践：**

1. **委派复杂任务**：使用子代理处理会混乱主代理上下文的多步骤工作。

2. **保持子代理响应简洁**：指示子代理返回摘要而非原始数据：

   ```python
   research_subagent = {
       "name": "researcher",
       "description": "Conducts research on a topic",
       "system_prompt": """You are a research assistant.
       IMPORTANT: Return only the essential summary (under 500 words).
       Do NOT include raw search results or detailed tool outputs.""",
       "tools": [web_search],
   }
   ```

3. **使用文件系统存储大数据**：子代理可以将结果写入文件；主代理读取所需内容。

## 长期记忆

使用默认文件系统时，你的 deep agent 将工作记忆文件存储在代理状态中，仅在单个线程内持久化。长期记忆使你的 deep agent 能够跨不同线程和对话持久化信息。Deep agents 可以使用长期记忆存储用户偏好、积累的知识、研究进度或应超出单个会话的任何信息。

要使用长期记忆，你必须使用 `CompositeBackend`，将特定路径（通常为 `/memories/`）路由到 LangGraph Store，后者提供持久的跨线程持久化。`CompositeBackend` 是一种混合存储系统，其中一些文件无限期持久化，而其他文件保持在单个线程范围内。

```python
from deepagents import create_deep_agent
from deepagents.backends import CompositeBackend, StateBackend, StoreBackend
from langgraph.store.memory import InMemoryStore

def make_backend(runtime):
    return CompositeBackend(
        default=StateBackend(runtime),
        routes={"/memories/": StoreBackend(runtime)},
    )

agent = create_deep_agent(
    model="google_genai:gemini-3.1-pro-preview",
    store=InMemoryStore(),
    backend=make_backend,
    system_prompt="""When users tell you their preferences, save them to
    /memories/user_preferences.txt so you remember them in future conversations.""",
)
```

你不需要预填充 `/memories/` 中的文件。你提供 backend 配置、store 和系统提示指令，告诉代理*保存什么*和*保存到哪里*。例如，你可以提示代理将偏好存储在 `/memories/preferences.txt`。路径开始时为空，代理在用户分享值得记住的信息时使用其文件系统工具（`write_file`、`edit_file`）按需创建文件。

要预填充记忆，在 LangSmith 上部署时使用 [Store API](/langsmith/custom-store)。

## 最佳实践

1. **从正确的输入上下文开始** – 保持 memory 最小化用于始终相关的约定；使用聚焦的 skills 获取任务特定能力。
2. **利用子代理处理繁重工作** – 委派多步骤、输出密集的任务以保持主代理上下文干净。
3. **在配置中调整子代理输出** – 如果调试时发现子代理生成长输出，可以在子代理的 `system_prompt` 中添加指导来创建摘要和综合发现。
4. **使用文件系统** – 将大输出持久化到文件（例如子代理写入或[自动卸载](#卸载)）以保持活动上下文小；模型可以在需要细节时用 `read_file` 和 `grep` 拉取片段。
5. **记录长期记忆结构** – 告诉代理 `/memories/` 中有什么以及如何使用它。
6. **为工具传递运行时上下文** – 使用 `context` 传递用户元数据、API keys 和工具需要的其他静态配置。
