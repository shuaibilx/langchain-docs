# Profiles

> 打包 Deep Agents 在选择模型时应用的按提供商和按模型默认值

**Harness profiles** 让你打包 Deep Agents 在选择给定提供商或特定模型时应用的配置：系统提示调整、工具描述覆盖、排除的工具或 middleware、额外 middleware，以及 general-purpose subagent 编辑。它们是调优特定模型 harness 行为的主要方式，无需更改 `create_deep_agent` 调用点。在 Python 中构建 profiles 时使用 `HarnessProfile`；[加载或保存 YAML/JSON 文件](#从配置文件加载-profiles)时使用 `HarnessProfileConfig`。Deep Agents 为 OpenAI 和 Anthropic（Claude）模型内置了 harness profiles。

**Provider profiles** 是一个更窄的配套 API，用于*模型构建* kwargs，不影响 harness。大多数调用者不需要它们；当你想要 `init_chat_model` 默认值、凭证检查或运行时派生的 kwargs 作为提供商选择的默认值时使用（例如打包提供商集成时）。

## Harness Profiles

`HarnessProfile` 描述了 `create_deep_agent` 在聊天模型构建后应用的提示组装、工具可见性、middleware 和默认 subagent 调整：

```python
from deepagents import (
    GeneralPurposeSubagentProfile,
    HarnessProfile,
    register_harness_profile,
)

register_harness_profile(
    "openai:gpt-5.4",
    HarnessProfile(
        system_prompt_suffix="Respond in under 100 words.",
        excluded_tools={"execute"},
        excluded_middleware={"SummarizationMiddleware"},
        general_purpose_subagent=GeneralPurposeSubagentProfile(enabled=False),
    ),
)
```

### 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `base_system_prompt` | `string` | 替换基础 Deep Agents 系统提示 |
| `system_prompt_suffix` | `string` | 追加到组装的基础提示末尾；应用于主代理、声明式 subagent 和自动添加的 general-purpose subagent |
| `tool_description_overrides` | `Mapping[str, str]` | 覆盖单个工具描述，按工具名键控 |
| `excluded_tools` | `frozenset[str]` | 从工具集中移除特定 harness 级工具。按工具名匹配，作为注入后过滤器应用 |
| `excluded_middleware` | `frozenset[type | str]` | 从栈中剥离特定 middleware 类。接受 middleware 类或字符串名 |
| `extra_middleware` | `Sequence[AgentMiddleware]` | 向此 profile 适用的每个栈追加 middleware |
| `general_purpose_subagent` | `GeneralPurposeSubagentProfile` | 禁用、重命名或重新提示 general-purpose subagent |

<Note>
  调用者提供的 `system_prompt=` 始终位于组装提示的前面，`system_prompt_suffix` 始终位于末尾——无论选择哪个模型。相同的覆盖规则适用于 subagent：每个 subagent 根据自己的模型重新运行 profile 解析。
</Note>

<Warning>
  要运行没有 `task` 工具的代理，参见[无 subagent 运行](/oss/python/deepagents/subagents#running-without-subagents)——设置 `general_purpose_subagent=GeneralPurposeSubagentProfile(enabled=False)` 并不通过 `subagents=` 传递同步 subagent。

  在 `excluded_middleware` 中列出 `FilesystemMiddleware`、`SubAgentMiddleware` 或内部权限 middleware 会引发 `ValueError`——它们是必需的脚手架。要从模型隐藏它们的工具而不移除 middleware，使用 `excluded_tools` 代替。
</Warning>

`excluded_middleware` 中的条目接受两种形式：

* Middleware *类*（按精确类型匹配），或匹配 `AgentMiddleware.name` 的纯字符串。
* `module:Class` 导入引用（例如 `"my_pkg.middleware:TelemetryMiddleware"`）来定位配置文件中的精确 middleware 类。

<Accordion title="预配置模型实例的查找顺序">
  当你传递预配置的聊天模型实例而不是 `provider:model` 字符串时，harness 从实例合成规范的 `provider:identifier` 键并按此顺序查找：

  1. 精确 `provider:identifier` 匹配
  2. 仅标识符（仅当标识符已包含 `:` 时）
  3. 仅提供商回退
</Accordion>

## 注册键

两种 profile 类型使用相同的键格式：

* **提供商级** — 裸提供商名如 `"openai"` 适用于该提供商的每个模型。
* **模型级** — 完全限定的 `provider:model` 键如 `"openai:gpt-5.4"` 仅适用于该特定模型。

当同时存在提供商级和模型级 profile 时，它们在解析时合并。未设置的模型级字段从提供商级 profile 继承；显式模型级值覆盖它们。

在现有键下重新注册将新 profile 合并到先前的 profile 上——不替换它。参见[合并语义](#合并语义)了解每字段规则。

<Note>
  没有匹配每个提供商的通配符键。要在各处应用相同的覆盖——比如无论选择哪个模型都删除 `TodoListMiddleware`——在你使用的每个提供商键下注册 profile。Profiles 旨在进行依赖于所选模型的调整。无论模型如何都应应用的全局调整应在 `create_deep_agent` 调用点上进行。
</Note>

## 合并语义

| 字段 | 合并行为 |
|------|---------|
| `base_system_prompt`、`system_prompt_suffix` | 设置时新值获胜；否则继承 |
| `tool_description_overrides` | 映射按键合并；共享键上新值获胜 |
| `excluded_tools`、`excluded_middleware` | 集合并 |
| `extra_middleware` | 按具体类合并：新实例替换其位置的现有实例，新类追加 |
| `general_purpose_subagent` | 按字段合并（未设置字段继承） |
| `init_kwargs`（provider） | 字典按键合并；共享键上新值获胜 |
| `pre_init`（provider） | 可调用链：现有先运行，然后新的 |
| `init_kwargs_factory`（provider） | 工厂链，每次 `resolve_model` 调用时输出合并 |

## Provider Profiles

`ProviderProfile` 声明 Deep Agents 应如何为给定提供商或特定模型规范构建聊天模型。它仅在你创建 deep agent 时提供 `provider:model` 字符串时应用，不适用于传递预配置模型时：

```python
from deepagents import ProviderProfile, register_provider_profile

register_provider_profile(
    "openai",
    ProviderProfile(init_kwargs={"temperature": 0}),
)
```

### 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `init_kwargs` | `Mapping[str, Any]` | 转发给 `init_chat_model` 的静态初始化参数 |
| `pre_init` | `Callable[[str], None]` | 构建前运行的副作用（例如凭证验证） |
| `init_kwargs_factory` | `Callable[[], dict[str, Any]]` | 从运行时状态派生的 kwargs（例如从环境变量获取的头） |

## 从配置文件加载 Profiles

对于 YAML/JSON 支持的工作流，使用 `HarnessProfileConfig`。它镜像 `HarnessProfile` 的声明式子集（提示文本、工具描述覆盖、排除的工具和 middleware、general-purpose subagent 编辑），并拥有 `to_dict` / `from_dict`。运行时状态——middleware 实例、工厂和类形式的 `excluded_middleware` 条目——保留在 `HarnessProfile` 上。

`register_harness_profile` 接受任一类型，因此配置支持的调用者不需要手动转换步骤：

```yaml
# openai.yaml
base_system_prompt: You are helpful.
system_prompt_suffix: Respond briefly.
excluded_tools:
  - execute
  - grep
excluded_middleware:
  - SummarizationMiddleware
  - my_pkg.middleware:TelemetryMiddleware
general_purpose_subagent:
  enabled: false
```

```python
import yaml
from deepagents import HarnessProfileConfig, register_harness_profile

with open("openai.yaml") as f:
    register_harness_profile(
        "openai",
        HarnessProfileConfig.from_dict(yaml.safe_load(f)),
    )
```

反向操作时，`HarnessProfileConfig.from_harness_profile(...)` 在仅使用可序列化功能时将运行时 profile 导出回声明式形状。

## 将 Profile 作为插件发布

可分发的 profiles 可以通过 `importlib.metadata` 入口点自行注册，而不需要调用者手动运行 `register_*_profile`。加载顺序是**内置优先，然后入口点插件，然后用户代码中的直接 `register_*_profile` 调用**；所有三个路径通过相同的加性注册，因此后续注册在相同键下分层在先前注册之上。

在分发的 `pyproject.toml` 中声明入口点：

```toml
[project.entry-points."deepagents.harness_profiles"]
my_provider = "my_pkg.profiles:register_harness"

[project.entry-points."deepagents.provider_profiles"]
my_provider = "my_pkg.profiles:register_provider"
```

每个目标解析为一个零参数可调用对象，在导入 `deepagents.profiles` 时执行注册：

```python
from deepagents import (
    HarnessProfile,
    ProviderProfile,
    register_harness_profile,
    register_provider_profile,
)


def register_harness() -> None:
    register_harness_profile(
        "my_provider",
        HarnessProfile(system_prompt_suffix="Batch independent tool calls in parallel."),
    )


def register_provider() -> None:
    register_provider_profile(
        "my_provider",
        ProviderProfile(init_kwargs={"temperature": 0}),
    )
```
