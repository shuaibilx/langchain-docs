# 与 Claude Agent SDK 的比较

> 比较 LangChain Deep Agents 和 Claude Agent SDK，为你的用例选择合适的工具。

本页解释 [LangChain Deep Agents](/oss/python/deepagents/overview) 与 [Claude Agent SDK](https://platform.anthropic.com/docs/en/agent-sdk/overview) 的比较。两者都是用于构建自定义代理的脚手架，但它们在执行环境、部署和供应商耦合方面做出了不同的权衡。

> Deep Agents 已被 [OpenSWE](https://github.com/langchain-ai/open-swe) 和 [LangSmith Fleet](/langsmith/fleet/index) 在生产环境中使用。

## 一览表

| | **Deep Agents** | **Claude Agent SDK** |
|---|---|---|
| **代理运行位置** | 在沙箱内，或在沙箱外远程执行命令 | 在沙箱内 |
| **执行后端** | 可插拔：[本地、虚拟文件系统、远程沙箱或自定义](/oss/python/deepagents/backends) | 运行所在沙箱的本地文件系统 |
| **模型提供商** | 任意（Anthropic、OpenAI、Google、100+ 其他） | Claude（Anthropic、Bedrock、Vertex、Azure） |
| **每提供商/模型调优** | [Harness profiles](/oss/python/deepagents/profiles)（beta）：声明式系统提示、工具、middleware 和子代理调整包，按提供商或特定模型注册 | 在每个模型调用点的代码中配置 |
| **部署** | LangSmith 中的 [Managed Deep Agents](/langsmith/deploy-managed-deep-agent)，或通过 [`langgraph build`](/langsmith/cli#build) 自托管[独立镜像](/langsmith/deploy-standalone-server) | [自托管](https://code.claude.com/docs/en/agent-sdk/hosting)。你需要构建服务器、认证和流式传输层。[Claude managed agents](https://platform.claude.com/docs/en/managed-agents/overview) 是独立产品 |
| **多租户** | [内置](/oss/python/deepagents/going-to-production#multi-tenancy)：范围线程、每用户沙箱、RBAC | 自行构建 |
| **许可证** | MIT | MIT（Claude Code 本身是专有的） |

## 主要区别

### 代理和执行环境

[连接代理到沙箱有两种模式](https://www.langchain.com/blog/the-two-patterns-by-which-agents-connect-sandboxes)：代理*在沙箱内*运行，或代理在外部运行并**将沙箱作为工具使用**。

Claude Agent SDK 仅支持第一种。你的代理在沙箱内运行，对沙箱的本地文件系统执行工具。Anthropic 的托管模型 [Claude managed agents](https://platform.claude.com/docs/en/managed-agents/overview) 使用解耦模型，这反映了生产代理架构的发展方向。

Deep Agents 两种都支持，并让你选择 [backend](/oss/python/deepagents/backends#quickstart) 来连接它们。在实践中，这意味着你可以：

* 在沙箱内运行代理（与 Claude Agent SDK 相同的模型）。
* 在长期运行的容器中运行代理并[将远程沙箱作为工具使用](https://www.langchain.com/blog/the-two-patterns-by-which-agents-connect-sandboxes)，通过网络执行命令。
* 为测试换入虚拟文件系统，或为你的基础设施换入自定义 backend。

### 多租户

当你将应用产品化时，通常会将其暴露给多个最终用户，并且必须为每个用户隔离环境。

在 Claude Agent SDK 中，SDK 将代理绑定到其沙箱。要为每个用户提供隔离的执行环境，你必须构建一个 API 包装器来为每个用户启动沙箱、跟踪哪个沙箱属于谁，并在之后销毁它。

Deep Agents 直接处理这个问题：在脚手架中[按用户或按助手](/oss/python/deepagents/going-to-production#lifecycle)配置沙箱，包含范围线程、运行历史和 [RBAC](/oss/python/deepagents/going-to-production#team-access-control-rbac)。如果你使用 [LangSmith Sandbox](/langsmith/sandbox-auth-proxy)，你还可以开箱即用地获得认证代理，这样最终用户可以从沙箱调用第三方 API，而你无需为每个用户配置凭据。

### 生产代理服务器

要将[自托管的 Claude Agent SDK](https://code.claude.com/docs/en/agent-sdk/hosting) 应用暴露给最终用户，你需要编写自己的 HTTP/WebSocket 或 SSE 服务器来调用代理、回传 token 流和管理对话线程。该服务器由你构建、运营和保护。

Deep Agents 部署开箱即用地包含一个[代理服务器](/langsmith/agent-server)：流式端点、线程管理、运行历史、webhooks 和[认证](/langsmith/auth)。

### 托管云或自托管

Claude Agent SDK 部署是[自托管的](https://code.claude.com/docs/en/agent-sdk/hosting)。SDK 和 [Claude managed agents](https://platform.claude.com/docs/en/managed-agents/overview) 是独立产品。针对 SDK 编写的代码不能直接部署到托管产品中。

Deep agents 以两种模式运行，无需更改代码：

* **托管：** 在 LangSmith 中使用 [Managed Deep Agents](/langsmith/deploy-managed-deep-agent) 创建、运行和操作 deep agents。
* **自托管：** 运行 [`langgraph build`](/langsmith/cli#build) 生成[独立 Docker 镜像](/langsmith/deploy-standalone-server)，你可以部署到任何地方。

> 对于跨任何模型提供商的托管代理平台，使用 [LangSmith Fleet](/langsmith/fleet/index)。[Claude managed agents](https://platform.claude.com/docs/en/managed-agents/overview) 仅限于 Anthropic 生态系统。

### LLM

Claude Agent SDK 执行捆绑了模型、后端和部署，并在三者之间优化支持。

使用 Deep Agents，你可以独立选择模型提供商、执行后端和部署目标。选择此脚手架，你在模型和基础设施选择方面保留最大灵活性。

### 生态系统

Claude Agent SDK 专为 Claude 和 Anthropic 的产品面而构建。Deep Agents 与更广泛的 LangChain 生态系统集成，包括用于可观测性、评估和部署的 LangSmith，并跨任何模型提供商工作。

## 总结

* **选择 Deep Agents**：如果你想要模型和基础设施灵活性、内置多租户部署，以及无需更改代码即可选择托管或自托管的选项。
* **选择 Claude Agent SDK**：如果你已投入 Anthropic 生态系统，并希望自托管并自行构建 API、认证和多租户层。
