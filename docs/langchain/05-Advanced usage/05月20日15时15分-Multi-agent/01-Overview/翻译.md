# 多智能体（Multi-agent）

多智能体系统协调专门化的组件来处理复杂的工作流。然而，并非每个复杂任务都需要这种方法——一个拥有正确（有时是动态的）工具和提示的单个 Agent 通常可以达到类似的结果。

> **提示：** 对于内置的多智能体支持，请使用 [Deep Agents](/oss/python/deepagents/overview)：一个基于 LangChain 构建的更高层工具，内置了[子 Agent](/oss/python/deepagents/subagents)、[技能](/oss/python/deepagents/skills)、规划、虚拟文件系统和上下文管理。

## 为什么需要多智能体？

当开发者说他们需要"多智能体"时，他们通常在寻找以下一种或多种能力：

* **上下文管理**：提供专门知识而不压倒模型的上下文窗口。如果上下文是无限的且延迟为零，你可以将所有知识倾倒到单个提示中——但由于事实并非如此，你需要模式来选择性地呈现相关信息。
* **分布式开发**：允许不同的团队独立开发和维护能力，并将它们组合成具有清晰边界的更大系统。
* **并行化**：为子任务生成专门的工作者并并发执行以获得更快的结果。

当单个 Agent 有太多[工具](/oss/python/langchain/tools)且在选择使用哪些工具时做出糟糕决策时，当任务需要具有大量上下文的专门知识（长提示和领域特定工具）时，或者当你需要强制执行只有在满足某些条件后才解锁能力的顺序约束时，多智能体模式特别有价值。

> **提示：** 多智能体设计的核心是**[上下文工程](/oss/python/langchain/context-engineering)**——决定每个 Agent 看到什么信息。你系统的质量取决于确保每个 Agent 能够访问其任务所需的正确数据。

## 模式

以下是构建多智能体系统的主要模式，每种模式适用于不同的用例：

| 模式 | 工作方式 |
|------|---------|
| [**子 Agent（Subagents）**](/oss/python/langchain/multi-agent/subagents) | 主 Agent 将子 Agent 作为工具来协调。所有路由都通过主 Agent，由主 Agent 决定何时以及如何调用每个子 Agent。 |
| [**切换（Handoffs）**](/oss/python/langchain/multi-agent/handoffs) | 行为根据状态动态变化。工具调用更新状态变量，触发路由或配置更改，切换 Agent 或调整当前 Agent 的工具和提示。 |
| [**技能（Skills）**](/oss/python/langchain/multi-agent/skills) | 按需加载的专门化提示和知识。单个 Agent 保持控制，同时根据需要从技能加载上下文。 |
| [**路由器（Router）**](/oss/python/langchain/multi-agent/router) | 路由步骤对输入进行分类并将其定向到一个或多个专门化的 Agent。结果被合成为组合响应。 |
| [**自定义工作流（Custom workflow）**](/oss/python/langchain/multi-agent/custom-workflow) | 使用 [LangGraph](/oss/python/langgraph/overview) 构建定制的执行流，混合确定性逻辑和 Agent 行为。将其他模式嵌入为工作流中的节点。 |

### 选择模式

使用此表将你的需求匹配到正确的模式：

| 模式 | 分布式开发 | 并行化 | 多跳 | 直接用户交互 |
|------|:---------:|:------:|:----:|:----------:|
| [**子 Agent**](/oss/python/langchain/multi-agent/subagents) | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐ |
| [**切换**](/oss/python/langchain/multi-agent/handoffs) | - | - | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| [**技能**](/oss/python/langchain/multi-agent/skills) | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| [**路由器**](/oss/python/langchain/multi-agent/router) | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | - | ⭐⭐⭐ |

* **分布式开发**：不同团队能否独立维护组件？
* **并行化**：多个 Agent 能否并发执行？
* **多跳**：该模式是否支持串联调用多个子 Agent？
* **直接用户交互**：子 Agent 能否直接与用户对话？

> **提示：** 你可以混合使用模式！例如，**子 Agent** 架构可以调用自定义工作流或路由器 Agent 的工具。子 Agent 甚至可以使用**技能**模式按需加载上下文。可能性是无穷的！

### 可视化概述

**子 Agent：** 主 Agent 将子 Agent 作为工具来协调。所有路由都通过主 Agent。

**切换：** Agent 通过工具调用相互传递控制权。每个 Agent 可以切换给其他 Agent 或直接响应用户。

**技能：** 单个 Agent 按需加载专门化的提示和知识，同时保持控制。

**路由器：** 路由步骤对输入进行分类并将其定向到专门化的 Agent。结果被合成。

> **提示：** 使用 [LangSmith](https://smith.langchain.com) 追踪跨 Agent 的完整协调流程。按照[追踪快速开始](/langsmith/trace-with-langchain)进行设置。

## 性能比较

不同的模式有不同的性能特征。理解这些权衡有助于你根据延迟和成本需求选择正确的模式。

**关键指标：**

* **模型调用**：LLM 调用次数。更多调用 = 更高延迟（特别是串行时）和更高的每次请求 API 成本。
* **处理的 token**：所有调用的总[上下文窗口](/oss/python/langchain/context-engineering)使用量。更多 token = 更高的处理成本和潜在的上下文限制。

### 单次请求

> **用户：** "买咖啡"

专门化的咖啡 Agent/技能可以调用 `buy_coffee` 工具。

| 模式 | 模型调用 | 最佳选择 |
|------|:-------:|:-------:|
| [**子 Agent**](/oss/python/langchain/multi-agent/subagents) | 4 | |
| [**切换**](/oss/python/langchain/multi-agent/handoffs) | 3 | ✅ |
| [**技能**](/oss/python/langchain/multi-agent/skills) | 3 | ✅ |
| [**路由器**](/oss/python/langchain/multi-agent/router) | 3 | ✅ |

**关键洞察：** 切换、技能和路由器对于单次任务最高效（各 3 次调用）。子 Agent 额外增加一次调用，因为结果通过主流回——这种开销提供了集中控制。

### 重复请求

> **第 1 轮：** "买咖啡"
> **第 2 轮：** "再买一次咖啡"

用户在同一对话中重复相同的请求。

| 模式 | 第 2 轮调用 | 总计（两轮） | 最佳选择 |
|------|:---------:|:----------:|:-------:|
| [**子 Agent**](/oss/python/langchain/multi-agent/subagents) | 4 | 8 | |
| [**切换**](/oss/python/langchain/multi-agent/handoffs) | 2 | 5 | ✅ |
| [**技能**](/oss/python/langchain/multi-agent/skills) | 2 | 5 | ✅ |
| [**路由器**](/oss/python/langchain/multi-agent/router) | 3 | 6 | |

**关键洞察：** 有状态模式（切换、技能）在重复请求上节省 40-50% 的调用。子 Agent 维持每次请求的一致成本——这种无状态设计提供了强上下文隔离，但代价是重复的模型调用。

### 多领域

> **用户：** "比较 Python、JavaScript 和 Rust 在 Web 开发方面的表现"

每个语言 Agent/技能包含约 2000 个 token 的文档。所有模式都可以进行并行工具调用。

| 模式 | 模型调用 | 总 token | 最佳选择 |
|------|:-------:|:-------:|:-------:|
| [**子 Agent**](/oss/python/langchain/multi-agent/subagents) | 5 | ~9K | ✅ |
| [**切换**](/oss/python/langchain/multi-agent/handoffs) | 7+ | ~14K+ | |
| [**技能**](/oss/python/langchain/multi-agent/skills) | 3 | ~15K | |
| [**路由器**](/oss/python/langchain/multi-agent/router) | 5 | ~9K | ✅ |

**关键洞察：** 对于多领域任务，具有并行执行的模式（子 Agent、路由器）最高效。技能调用次数较少但 token 使用量高，因为上下文累积。切换在这里效率低下——它必须串行执行，无法利用并行工具调用来同时查询多个领域。

### 总结

以下是各模式在所有三个场景中的比较：

| 模式 | 单次请求 | 重复请求 | 多领域 |
|------|:-------:|:-------:|:-----:|
| [**子 Agent**](/oss/python/langchain/multi-agent/subagents) | 4 次调用 | 8 次调用 (4+4) | 5 次调用，9K token |
| [**切换**](/oss/python/langchain/multi-agent/handoffs) | 3 次调用 | 5 次调用 (3+2) | 7+ 次调用，14K+ token |
| [**技能**](/oss/python/langchain/multi-agent/skills) | 3 次调用 | 5 次调用 (3+2) | 3 次调用，15K token |
| [**路由器**](/oss/python/langchain/multi-agent/router) | 3 次调用 | 6 次调用 (3+3) | 5 次调用，9K token |

**选择模式：**

| 优化方向 | 子 Agent | 切换 | 技能 | 路由器 |
|---------|:-------:|:----:|:---:|:-----:|
| 单次请求 | | ✅ | ✅ | ✅ |
| 重复请求 | | ✅ | ✅ | |
| 并行执行 | ✅ | | | ✅ |
| 大上下文领域 | ✅ | | | ✅ |
| 简单、专注的任务 | | | ✅ | |
