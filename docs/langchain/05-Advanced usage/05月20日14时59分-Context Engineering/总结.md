# Context Engineering 文档总结

## 一句话概述

上下文工程是在正确的时间以正确的格式向 LLM 提供正确的信息和工具，这是构建可靠 Agent 的首要任务。

---

## Agent 失败的两大原因

1. LLM 能力不够
2. **"正确的"上下文没有传递给 LLM** ← 更常见

---

## 三种上下文类型

| 类型 | 你控制什么 | 瞬时/持久 |
|------|-----------|----------|
| **模型上下文** | 系统提示、消息、工具、模型、响应格式 | 瞬时（每次调用） |
| **工具上下文** | 工具对 State/Store/Runtime 的读写 | 持久 |
| **生命周期上下文** | 模型和工具调用之间的事（总结、护栏等） | 持久 |

---

## 三种数据源

| 数据源 | 也称为 | 作用域 | 示例 |
|-------|-------|-------|------|
| **Runtime Context** | 静态配置 | 对话级 | 用户ID、API密钥、权限 |
| **State** | 短期记忆 | 对话级 | 消息、文件、认证状态 |
| **Store** | 长期记忆 | 跨对话 | 用户偏好、历史数据 |

---

## 模型上下文的 5 个维度

### 1. System Prompt
- 从 State 获取：对话长度 → 调整风格
- 从 Store 获取：用户偏好 → 个性化
- 从 Runtime 获取：角色/环境 → 权限控制

### 2. Messages
- 从 State 注入：上传文件上下文
- 从 Store 注入：写作风格
- 从 Runtime 注入：合规规则

### 3. Tools
- 从 State 过滤：认证状态 → 工具可见性
- 从 Store 过滤：功能标志
- 从 Runtime 过滤：角色权限

### 4. Model
- 从 State 选择：对话长度 → 模型大小
- 从 Store 选择：用户偏好模型
- 从 Runtime 选择：成本/环境层级

### 5. Response Format
- 从 State 选择：对话阶段 → 简单/详细
- 从 Store 选择：用户偏好风格
- 从 Runtime 选择：角色 → 输出详细度

## 工具上下文：读写

| 操作 | State | Store | Runtime |
|------|-------|-------|---------|
| 读取 | `runtime.state` | `runtime.store.get()` | `runtime.context` |
| 写入 | `Command(update=...)` | `runtime.store.put()` | — |

---

## 生命周期上下文

核心机制：中间件（Middleware）

```
模型调用前 → before_model
模型调用后 → after_model
工具调用前 → before_tool
工具调用后 → after_tool
Agent 开始前 → before_agent
Agent 完成后 → after_agent
```

关键操作：
1. **更新上下文** — 修改 State/Store
2. **生命周期跳转** — 跳到不同步骤

---

## 瞬时 vs 持久

| 类型 | 机制 | 效果 |
|------|------|------|
| **瞬时** | `wrap_model_call` + `request.override()` | 单次调用修改，不影响 State |
| **持久** | 返回 `Command` 或使用生命周期钩子 | 永久修改 State |

## 重点与难点

### 重点
- 三种上下文类型（模型/工具/生命周期）的区别
- 三种数据源（Runtime/State/Store）的作用域
- 瞬时 vs 持久更新的机制

### 难点
- 何时用瞬时更新 vs 持久更新
- State/Store/Runtime 三者的读写方式不同
- 动态工具选择的实现模式
- 生命周期跳转的使用场景

---

## 学习建议

1. **先理解三种数据源**——Runtime/State/Store 的区别是一切的基础
2. **从模型上下文开始**——System Prompt 和 Messages 是最常用的
3. **理解瞬时 vs 持久**——这是上下文工程的核心区分
4. **工具上下文是桥梁**——工具连接了模型和外部世界
5. **生命周期上下文是高级特性**——需要时再深入
