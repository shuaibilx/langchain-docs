# Guardrails 文档总结

## 一句话概述

安全护栏通过中间件在 Agent 执行的关键节点拦截和验证内容，构建安全合规的 AI 应用。

---

## 两种实现方式

| 方式 | 特点 | 适用场景 |
|------|------|----------|
| **确定性护栏** | 规则、正则、关键词匹配 | 快速、可预测、低成本 |
| **基于模型的护栏** | LLM/分类器评估 | 语义理解、捕获微妙违规 |

---

## 内置护栏

### PII 检测

| 策略 | 效果 |
|------|------|
| `redact` | `[REDACTED_EMAIL]` |
| `mask` | `****-****-****-1234` |
| `hash` | `a8f5f167...` |
| `block` | 抛出异常 |

关键参数：
- `apply_to_input` — 检查输入（默认 True）
- `apply_to_output` — 检查输出（默认 False）
- `apply_to_tool_results` — 检查工具结果（默认 False）

### 人类在环

- `interrupt_on={"tool_name": True/False}` 控制哪些工具需要审批
- 需要 `checkpointer` + `thread_id` 持久化状态
- 用 `Command(resume={"decisions": [...]})` 恢复执行

---

## 自定义护栏

### Before Agent 护栏
- 时机：每次调用开始时（一次）
- 用途：认证、限流、内容过滤
- 钩子：`before_agent` + `can_jump_to=["end"]`

### After Agent 护栏
- 时机：返回用户前（一次）
- 用途：输出质量验证、合规扫描
- 钩子：`after_agent` + `can_jump_to=["end"]`

---

## 分层防护架构

```
请求进入
  ↓
第 1 层：确定性输入过滤（before_agent）
  ↓
第 2 层：PII 保护（before_model）
  ↓
第 3 层：人类审批（wrap_tool_call）
  ↓
第 4 层：模型安全检查（after_agent）
  ↓
响应返回
```

---

## 与 Custom Middleware 的关系

Guardrails = Custom Middleware 的**安全应用场景**。核心思想完全相同：
- 用 `before_agent` 做输入验证
- 用 `after_agent` 做输出验证
- 用 `wrap_*` 做过程控制
- 用 `can_jump_to` 提前终止

---

## 重点与难点

### 重点
- PII 的 4 种策略（redact/mask/hash/block）
- HITL 的 interrupt_on 配置
- 分层防护的组合模式

### 难点
- 确定性 vs 基于模型护栏的选择
- PII 检测的 apply_to 三个方向
- 安全检查的性能影响（模型调用增加延迟）
- 护栏误判的处理

---

## 学习建议

1. **PII 是最实用的内置护栏**——几乎每个生产系统都需要
2. **HITL 是最安全的护栏**——高风险操作必须有
3. **分层防护是最佳实践**——不要依赖单一护栏
4. **确定性优先**——能用规则解决的不用模型
