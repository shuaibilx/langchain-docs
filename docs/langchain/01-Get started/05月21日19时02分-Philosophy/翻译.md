# Philosophy

> LangChain 的存在是为了成为使用 LLM 构建的最简单起点，同时保持灵活性和生产就绪。

LangChain 由几个核心信念驱动：

- 大语言模型（LLM）是伟大、强大的新技术。
- 当你将 LLM 与外部数据源结合时，它们会更好。
- LLM 将改变未来应用的样子。具体来说，未来的应用将越来越具有 agent 特性。
- 在那场变革中，现在还非常早期。
- 虽然构建那些 agent 应用的原型很容易，但构建足够可靠以投入生产的 agent 仍然非常困难。

今天开发者可以选择如何构建 agent：使用 [LangChain](/oss/python/langchain/overview) 获得最大的灵活性和控制力，或使用 [Deep Agents](/oss/python/langchain/overview)，它提供类似的灵活性和控制力，但附带了有主见的内置规划、文件系统工具、子 agent 和上下文管理。两者都构建在 [LangGraph](/oss/python/langgraph/overview) 之上。

使用 LangChain，我们有两个核心焦点：

### 1. 我们想要帮助开发者使用最好的模型进行构建

不同的提供商暴露不同的 API，具有不同的模型参数和不同的消息格式。标准化这些模型输入和输出是一个核心焦点，使开发者能够轻松切换到最新的最先进模型，避免锁定。

### 2. 我们想要帮助开发者使用模型来与其他数据和计算进行交互，编排更复杂的流程

模型应该被用于不仅仅是*文本生成*——它们还应该被用来编排与其他数据交互的更复杂流程。LangChain 使定义 LLM 可以动态使用的[工具](/oss/python/langchain/tools)变得容易，同时也帮助解析和访问非结构化数据。

## 历史

鉴于该领域的持续变化，LangChain 也随时间演变。以下是 LangChain 多年来如何演变的简要时间线：

### 2022-10-24 — v0.0.1

在 ChatGPT 发布前一个月，**LangChain 作为 Python 包发布**。它由两个主要组件组成：

- LLM 抽象
- "Chains"，或预先确定的计算步骤，用于常见用例。例如 - RAG：运行检索步骤，然后运行生成步骤。

LangChain 的名字来自 "Language"（像语言模型）和 "Chains"。

### 2022-12

第一批通用 agent 被添加到 LangChain。

这些通用 agent 基于 [ReAct 论文](https://arxiv.org/abs/2210.03629)（ReAct 代表推理和行动）。它们使用 LLM 生成代表工具调用的 JSON，然后解析该 JSON 来确定要调用什么工具。

### 2023-01

OpenAI 发布了 "Chat Completion" API。

以前，模型接收字符串并返回字符串。在 ChatCompletions API 中，它们演变为接收消息列表并返回一条消息。其他模型提供商效仿，LangChain 更新为使用消息列表。

### 2023-01

LangChain 发布了 JavaScript 版本。

LLM 和 agent 将改变应用的构建方式，而 JavaScript 是应用开发者的语言。

### 2023-02

**LangChain Inc. 作为公司**围绕开源 LangChain 项目成立。

主要目标是"使智能 agent 无处不在"。团队认识到虽然 LangChain 是一个关键部分（LangChain 使开始使用 LLM 变得简单），但也需要其他组件。

### 2023-03

OpenAI 在其 API 中发布了 "function calling"。

这允许 API 显式生成代表工具调用的载荷。其他模型提供商效仿，LangChain 更新为使用此方法作为工具调用的首选方法（而不是解析 JSON）。

### 2023-06

**LangSmith 作为闭源平台由 LangChain Inc. 发布**，提供可观测性和评估。

构建 agent 的主要问题是使它们可靠，LangSmith 提供可观测性和评估，就是为了满足这一需求而构建的。LangChain 更新为与 LangSmith 无缝集成。

### 2024-01 — v0.1.0

**LangChain 发布 0.1.0**，其第一个非 0.0.x 版本。

行业从原型成熟到生产，因此 LangChain 增加了对稳定性的关注。

### 2024-02

**LangGraph 作为开源库发布**。

原始 LangChain 有两个焦点：LLM 抽象和用于常见应用的高级接口；然而，它缺少一个允许开发者控制 agent 精确流程的低级编排层。Enter：LangGraph。

在构建 LangGraph 时，我们从构建 LangChain 时的经验中学习，并添加了我们发现需要的功能：流式传输、持久化执行、短期记忆、human-in-the-loop 等。

### 2024-06

**LangChain 拥有超过 700 个集成。**

集成从核心 LangChain 包中分离出来，要么移入自己的独立包（用于核心集成），要么移入 `langchain-community`。

### 2024-10

LangGraph 成为构建任何超过单次 LLM 调用的 AI 应用的首选方式。

随着开发者试图提高应用的可靠性，他们需要比高级接口提供的更多控制。LangGraph 提供了那种低级灵活性。大多数 chains 和 agents 在 LangChain 中被标记为已弃用，并附有关于如何将它们迁移到 LangGraph 的指南。LangGraph 中仍然创建了一个高级抽象：agent 抽象。它构建在低级 LangGraph 之上，与 LangChain 中的 ReAct agents 具有相同的接口。

### 2025-04

模型 API 变得更多模态。

模型开始接受文件、图像、视频等。我们相应更新了 `langchain-core` 消息格式，允许开发者以标准方式指定这些多模态输入。

### 2025-10-20 — v1.0.0

**LangChain 发布 1.0**，有两个重大变化：

1. **完全改造 `langchain` 中的所有 chains 和 agents。** 所有 chains 和 agents 现在只用一个高级抽象替代：一个构建在 LangGraph 之上的 agent 抽象。这是最初在 LangGraph 中创建的高级抽象，只是移到了 LangChain。

   对于仍在使用旧 LangChain chains/agents 且不想升级的用户（注意：我们建议你升级），你可以通过安装 `langchain-classic` 包继续使用旧 LangChain。

2. **标准消息内容格式：** 模型 API 从返回带有简单内容字符串的消息演变为更复杂的输出类型——推理块、引用、服务器端工具调用等。LangChain 演变了其消息格式以跨提供商标准化这些。

### 2026-03-15 — v0.5.3

**Deep Agents 作为构建在 LangGraph 之上的开源 agent harness 发布。**

虽然 LangChain 为自定义 agent 架构提供了灵活的构建块，但 [Deep Agents](/oss/python/langchain/overview) 为复杂的长时间运行任务（如研究和编码）提供了开箱即用的选项。它添加了内置规划工具、具有可插拔后端（内存、磁盘、LangGraph store、沙箱）的虚拟文件系统，以及用于上下文隔离的子 agent 生成。对于更自主的 agent 使用 Deep Agents 并预定义工具；对于完全控制 agent 架构使用 LangChain。
