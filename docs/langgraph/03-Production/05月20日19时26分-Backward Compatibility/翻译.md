# 向后兼容性（Backward Compatibility）

> 在生产环境中更新 LangGraph 图代码而不中断正在运行的任务。

软件在生产环境中需要改变。新需求、Bug 修复和重构最终都会进入你的图代码。因为 LangGraph 对已为现有线程[持久化](/oss/python/langgraph/persistence)的状态运行最新部署的图，你发布的每个更改实际上都是相对于现有检查点的向后兼容 API 变更。

与将运行固定到其启动时代码版本的工作流引擎不同，LangGraph 立即将最新图应用于*每个*线程，包括新线程和从检查点恢复的线程。这很方便：Bug 修复会自动传播到正在运行的对话和代理。这也意味着你必须推理每个更改如何与在先前代码版本下启动的运行交互。

有三类兼容性问题需要注意，大致按你遇到的顺序排列：

1. [技术兼容性](#技术兼容性)：最常见；新代码必须仍然能加载和执行现有状态。
2. [业务兼容性](#业务兼容性)：较少见；现有运行应继续遵循旧业务逻辑，即使代码已更改。
3. [非确定性](#非确定性)：仅适用于 [Functional API](/oss/python/langgraph/functional-api)。

## 技术兼容性

技术兼容性相当于微服务中的 API 破坏性变更。这里的"API"是你的图代码与[检查点器](/oss/python/langgraph/persistence#检查点器库)为现有线程持久化的数据之间的契约。当线程恢复时，LangGraph 反序列化保存的状态，按名称将其分派到节点，并期望节点返回适合状态 schema 的值。

常见的技术破坏：

* **重命名或删除节点**，而线程暂停在该节点或即将进入该节点。恢复时，LangGraph 无法按保存的名称找到节点，运行失败。
* **重命名或删除状态键**，旧检查点仍包含或下游节点仍读取该键。
* **收紧状态字段**，如将 `Optional` 字段设为必需、收窄类型或添加无默认值的新必需字段。

边拓扑本身*不*持久化在检查点中。在仍然存在的节点之间添加、删除或重路由边对正在运行的线程是安全的。唯一可能中断正在运行线程的拓扑变更时重命名或删除节点。

### 推荐模式

* 将新状态字段添加为 `NotRequired`（或 `Optional[...] = None`）：

```python
from typing import NotRequired
from typing_extensions import TypedDict

class State(TypedDict):
    messages: list
    summary: NotRequired[str]  # 新增字段
```

* 将删除视为弃用。至少在一个 drain 周期内保留字段定义。

* 通过*先添加后删除*进行重命名。在旧字段旁添加新字段，双写一段时间，确认无线程依赖后删除旧字段。

* 保持节点函数对未知键的容忍。`TypedDict` 在运行时忽略额外键。

* 使用[时间旅行](/oss/python/langgraph/use-time-travel)和 `graph.get_state` 在暂存部署中检查现有线程。

### 检测正在运行的线程

在删除节点或重命名状态键之前，你想知道是否有线程停在你即将删除的代码版本上。

**部署到 LangSmith 时**：使用 Agent Server 的线程搜索按状态过滤。`status` 字段接受 `idle`、`busy`、`interrupted` 和 `error`。

**任何运行 LangGraph 的地方**：使用 LangSmith tracing 监控生产中进入和退出的节点。

**已有 `thread_id` 时**：直接检查该线程：

```python
state = graph.get_state(config)        # 最新检查点
history = graph.get_state_history(config)  # 完整历史
```

## 业务兼容性

有时更改在技术上有效（每个现有检查点仍加载，每个节点仍解析），但新图的*含义*与旧图不同。新行为对新线程是正确的，你不想将其追溯应用于在旧逻辑下启动的线程。

例如，你的图运行 `intake → triage → respond`，你决定在 `triage` 和 `respond` 之间插入新的 `policy_check` 步骤：

* 已经过 `triage` 的线程应直接继续到 `respond`（旧流程）。
* 新线程应运行完整的新流程。

推荐模式是在**线程开始时**在状态上记录相关的*行为版本*，然后用[条件边](/oss/python/langgraph/graph-api#条件边)进行分支：

```python
from typing import NotRequired
from typing_extensions import TypedDict
from langgraph.graph import END, START, StateGraph

class State(TypedDict):
    request: str
    flow_version: NotRequired[int]
    response: NotRequired[str]

def intake(state: State) -> dict:
    # 为新线程打上当前流程版本。恢复过 intake 的现有线程保留已保存的值。
    return {"flow_version": state.get("flow_version", 2)}

def triage(state: State) -> dict: ...
def policy_check(state: State) -> dict: ...
def respond(state: State) -> dict: ...

def after_triage(state: State) -> str:
    if state.get("flow_version", 1) >= 2:
        return "policy_check"
    return "respond"

builder = StateGraph(State)
builder.add_node("intake", intake)
builder.add_node("triage", triage)
builder.add_node("policy_check", policy_check)
builder.add_node("respond", respond)
builder.add_edge(START, "intake")
builder.add_edge("intake", "triage")
builder.add_conditional_edges("triage", after_triage, ["policy_check", "respond"])
builder.add_edge("policy_check", "respond")
builder.add_edge("respond", END)

graph = builder.compile()
```

旧线程从保存的状态读取 `flow_version`（或回退到 v1 默认值）并跳过 `policy_check`。新线程从 `intake` 开始，被打上 `flow_version=2`，运行新路径。

此模式仅在你在*线程开始时*设置版本才有效。之后设置意味着现有线程在需要时不会有该值。

## 非确定性

此类别仅适用于 [Functional API](/oss/python/langgraph/functional-api)。[Graph API](/oss/python/langgraph/graph-api) 在恢复时在节点边界重新进入，因此节点代码不会像 Temporal 风格的工作流那样从函数开头"重放"。

Functional API 在运行恢复时从头重放 `@entrypoint` 的主体，使用缓存的 [`@task`](https://reference.langchain.com/python/langgraph/func/task) 结果跳过已完成的工作。两类更改会破坏此模型：

* **添加、删除或重排序位于恢复点之前的 `@task` 调用或 `interrupt` 调用**。LangGraph 按重放中的位置匹配缓存结果和恢复值。
* **在 `@task` 之外引入非确定性操作**，如 `time.time()`、`random.random()` 或内联在 entrypoint 主体中的网络调用。

如果需要对有正在运行的 `@entrypoint` 进行非平凡代码更改，最安全的选择是：

* 让正在运行的任务在部署更改前排空。
* 将新逻辑包装在新的 `@task` 中。
* 在 `langgraph.json` 中注册新的 entrypoint，将新线程路由到它。
