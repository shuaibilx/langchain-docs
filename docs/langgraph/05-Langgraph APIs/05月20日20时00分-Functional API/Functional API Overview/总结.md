# Functional API 概述 - 总结

## 核心定位

Functional API 是 LangGraph 的命令式编程接口，允许用标准 Python 代码（if/for/函数调用）构建工作流，无需声明图结构。

## 两个核心装饰器

| 装饰器 | 作用 | 特性 |
|--------|------|------|
| `@entrypoint` | 标记工作流入口函数 | 必须接受单个参数；支持 checkpointer/store/writer/config 注入 |
| `@task` | 封装离散工作单元 | 返回 future 对象；结果自动检查点；只能在 entrypoint 内调用 |

## Functional API vs Graph API

| 维度 | Functional API | Graph API |
|------|---------------|-----------|
| 控制流 | 标准 Python 语法 | 声明式图结构 |
| 状态管理 | 函数作用域，无需 reducers | 需声明 State + reducers |
| 检查点 | task 结果保存到现有检查点 | 每个 superstep 新检查点 |
| 可视化 | 不支持（运行时动态生成） | 支持图形可视化 |

## 关键概念

### Entrypoint
- 用 `@entrypoint(checkpointer=...)` 装饰
- 产生 `Pregel` 对象，支持 invoke/stream/ainvoke/astream
- 可注入参数：`previous`（短期记忆）、`store`（长期记忆）、`writer`（流式）、`config`（配置）

### Task
- 用 `@task` 装饰
- 返回 future 对象，需 `.result()` 或 `await` 获取结果
- 用途：检查点、人机交互、并行执行、可观测性、可重试

### 短期记忆
- `previous` 参数访问上一次调用的返回值
- `entrypoint.final[return_type, save_type]` 解耦返回值和保存值

### 确定性
- 随机性操作必须封装在 task 中
- 恢复时遵循相同的步骤序列
- 基于索引匹配 interrupt 和 resume 值

### 幂等性
- API 调用应设计为幂等的
- 使用幂等性键或验证现有结果

## 常见陷阱

1. **副作用**：必须封装在 task 中，避免重复执行
2. **非确定性控制流**：时间/随机数等操作必须放入 task

## 执行与恢复

```python
# 执行
workflow.invoke(input, config)
workflow.stream(input, config)

# 中断后恢复
workflow.stream(Command(resume=value), config)

# 错误后恢复
workflow.invoke(None, config)
```
