# 持久执行（Durable Execution）

**持久执行**是一种技术，进程或工作流在关键点保存进度，允许暂停后精确恢复到离开的位置。这在需要[人机交互](/oss/python/langgraph/interrupts)的场景中特别有用，用户可以在继续之前检查、验证或修改流程，也适用于可能遇到中断或错误的长时间运行任务（例如 LLM 调用超时）。通过保留已完成的工作，持久执行使流程能够在不重新处理先前步骤的情况下恢复——即使在显著延迟之后（例如一周后）。

LangGraph 内置的[持久化](/oss/python/langgraph/persistence)层为工作流提供持久执行，确保每个执行步骤的状态保存到持久存储中。此能力保证如果工作流被中断——无论是系统故障还是[人机交互](/oss/python/langgraph/interrupts)——都可以从最后记录的状态恢复。

> **提示：** 如果你正在使用带检查点器的 LangGraph，你已经启用了持久执行。你可以在任何时候暂停和恢复工作流，即使在中断或故障之后。
> 为了充分利用持久执行，确保你的工作流设计为[确定性](#确定性与一致重放)和[幂等性](#确定性与一致重放)的，并将任何副作用或非确定性操作包装在[任务](/oss/python/langgraph/functional-api#task)中。你可以在 [StateGraph（Graph API）](/oss/python/langgraph/graph-api)和 [Functional API](/oss/python/langgraph/functional-api)中使用任务。

## 要求

要在 LangGraph 中利用持久执行，你需要：

1. 通过指定[检查点器](/oss/python/langgraph/persistence#检查点器库)在工作流中启用[持久化](/oss/python/langgraph/persistence)，该检查点器将保存工作流进度。

2. 在执行工作流时指定[线程标识符](/oss/python/langgraph/persistence#线程)。这将跟踪特定工作流实例的执行历史。

3. 将任何非确定性操作（例如随机数生成）或有副作用的操作（例如文件写入、API 调用）包装在[任务](https://reference.langchain.com/python/langgraph/func/task)中，以确保当工作流恢复时，这些操作不会被重复执行，而是从持久层检索其结果。更多信息请参阅[确定性与一致重放](#确定性与一致重放)。

## 确定性与一致重放

当你恢复工作流运行时，代码**不会**从执行停止的**同一行代码**恢复；相反，它将识别一个适当的[起点](#恢复工作流的起点)来接续。这意味着工作流将从[起点](#恢复工作流的起点)重放所有步骤，直到到达停止的位置。

因此，当你为持久执行编写工作流时，必须将任何非确定性操作（例如随机数生成）和任何有副作用的操作（例如文件写入、API 调用）包装在[任务](/oss/python/langgraph/functional-api#task)或[节点](/oss/python/langgraph/graph-api#节点)中。

为确保你的工作流是确定性的并且可以一致重放，请遵循以下准则：

* **避免重复工作**：如果一个[节点](/oss/python/langgraph/graph-api#节点)包含多个有副作用的操作（例如日志记录、文件写入或网络调用），将每个操作包装在单独的**任务**中。这确保当工作流恢复时，操作不会被重复，其结果从持久层检索。

* **封装非确定性操作**：将任何可能产生非确定性结果的代码（例如随机数生成）包装在**任务**或**节点**中。这确保恢复时，工作流遵循记录的精确步骤序列和相同结果。

* **使用幂等操作**：在可能的情况下，确保副作用（例如 API 调用、文件写入）是幂等的。这意味着如果操作在工作流故障后重试，它将与第一次执行具有相同效果。这对于导致数据写入的操作特别重要。在**任务**开始但未能成功完成的情况下，工作流的恢复将重新运行**任务**，依赖记录的结果来维持一致性。使用幂等性键或验证现有结果以避免意外重复，确保顺畅可预测的工作流执行。

关于需要避免的陷阱示例，请参阅 Functional API 中的[常见陷阱](/oss/python/langgraph/functional-api#常见陷阱)部分，该部分展示了如何使用**任务**来构建代码以避免这些问题。相同原则适用于 [StateGraph（Graph API）](https://reference.langchain.com/python/langgraph/graph/state/StateGraph)。

## 持久性模式

LangGraph 支持三种持久性模式，允许你根据应用程序需求平衡性能和数据一致性。更高的持久性模式为工作流执行增加更多开销。你可以在调用任何图执行方法时指定持久性模式：

```python
graph.stream(
    {"input": "test"},
    durability="sync"
)
```

持久性模式从最低到最高持久性如下：

* `"exit"`：LangGraph 仅在图执行退出时持久化更改——无论是成功退出、出错还是由于人机交互中断。这为长时间运行的图提供最佳性能，但意味着中间状态不会保存，因此你无法从执行过程中发生的系统故障（如进程崩溃）中恢复。

* `"async"`：LangGraph 在下一步执行时异步持久化更改。这提供良好的性能和持久性，但如果进程在执行过程中崩溃，LangGraph 可能不会写入检查点，存在小风险。

* `"sync"`：LangGraph 在下一步开始前同步持久化更改。这确保 LangGraph 在继续执行前写入每个检查点，以一些性能开销为代价提供高持久性。

## 在节点中使用任务

如果一个[节点](/oss/python/langgraph/graph-api#节点)包含多个操作，你可能发现将每个操作转换为**任务**比将操作重构为单独的节点更容易。

**原始代码：**

```python
from typing import NotRequired
from typing_extensions import TypedDict
from langchain_core.utils.uuid import uuid7
from langgraph.checkpoint.memory import InMemorySaver
from langgraph.graph import StateGraph, START, END
import requests

class State(TypedDict):
    url: str
    result: NotRequired[str]

def call_api(state: State):
    result = requests.get(state['url']).text[:100]  # 有副作用
    return {"result": result}

builder = StateGraph(State)
builder.add_node("call_api", call_api)
builder.add_edge(START, "call_api")
builder.add_edge("call_api", END)

checkpointer = InMemorySaver()
graph = builder.compile(checkpointer=checkpointer)

thread_id = str(uuid7())
config = {"configurable": {"thread_id": thread_id}}
graph.invoke({"url": "https://www.example.com"}, config)
```

**使用 task 重构：**

```python
from typing import NotRequired
from typing_extensions import TypedDict
from langchain_core.utils.uuid import uuid7
from langgraph.checkpoint.memory import InMemorySaver
from langgraph.func import task
from langgraph.graph import StateGraph, START, END
import requests

class State(TypedDict):
    urls: list[str]
    result: NotRequired[list[str]]

@task
def _make_request(url: str):
    return requests.get(url).text[:100]  # 副作用封装在 task 中

def call_api(state: State):
    requests = [_make_request(url) for url in state['urls']]
    results = [request.result() for request in requests]
    return {"results": results}

builder = StateGraph(State)
builder.add_node("call_api", call_api)
builder.add_edge(START, "call_api")
builder.add_edge("call_api", END)

checkpointer = InMemorySaver()
graph = builder.compile(checkpointer=checkpointer)

thread_id = str(uuid7())
config = {"configurable": {"thread_id": thread_id}}
graph.invoke({"urls": ["https://www.example.com"]}, config)
```

## 恢复工作流

一旦你在工作流中启用了持久执行，你可以在以下场景中恢复执行：

* **暂停和恢复工作流**：使用 [interrupt](https://reference.langchain.com/python/langgraph/types/interrupt) 函数在特定点暂停工作流，使用 [`Command`](https://reference.langchain.com/python/langgraph/types/Command) 原语以更新的状态恢复。详见 [**中断**](/oss/python/langgraph/interrupts)。

* **从故障中恢复**：在异常（例如 LLM 提供商中断）后从最后一个成功检查点自动恢复工作流。这涉及使用相同线程标识符执行工作流，提供 `None` 作为输入值（参见 Functional API 中的[示例](/oss/python/langgraph/use-functional-api#resuming-after-an-error)）。

## 恢复工作流的起点

* 如果你使用 [StateGraph（Graph API）](/oss/python/langgraph/graph-api)，起点是执行停止的[**节点**](/oss/python/langgraph/graph-api#节点)的开始。

* 如果你在节点内进行子图调用，起点将是调用被停止子图的**父**节点。在子图内部，起点将是执行停止的特定[**节点**](/oss/python/langgraph/graph-api#节点)。

* 如果你使用 Functional API，起点是执行停止的 [**entrypoint**](/oss/python/langgraph/functional-api#entrypoint) 的开始。

## 优雅关闭

> 需要 `langgraph>=1.2`。

优雅关闭让你协作地停止正在运行的图执行——在当前超级步骤完成后——并保存可恢复的检查点。这对于处理 SIGTERM 信号或任何需要回收资源而不丢失工作的外部监管器很有用。

创建 [`RunControl`](https://reference.langchain.com/python/langgraph/runtime/RunControl) 并将其作为 `control=` 传递给 `invoke` 或 `stream`。从任何线程调用 `request_drain()` 以信号通知运行应该停止：

```python
from langgraph.runtime import RunControl
from langgraph.errors import GraphDrained

control = RunControl()

# 在信号处理器或监管器中：
# control.request_drain("sigterm")

try:
    result = graph.invoke(inputs, config, control=control)
except GraphDrained as e:
    # 图提前停止并保存了检查点。
    # 稍后使用相同配置恢复。
    print(f"Drained: {e.reason}")
```

### 语义

Drain 是协作的，在超级步骤之间操作，从不抢占正在运行的工作：

| 场景 | 行为 |
|------|------|
| 节点执行中 | 运行到完成。Drain 在下一个超级步骤生效。 |
| 节点正在重试策略中重试 | 重试循环运行到耗尽或成功。之后 Drain 生效。 |
| 图在同一 tick 自然完成 | 正常返回。检查 `control.drain_requested` 以区分正常运行。 |
| 还有更多超级步骤 | 抛出 `GraphDrained(reason)`。检查点已保存，可恢复。 |
| 子图请求 drain | `GraphDrained` 冒泡到父图，在其自身的下一个超级步骤边界停止。 |

### Drain 后恢复

使用相同 `thread_id` 的 `invoke(None, config)` 恢复 drained 的运行：

```python
result = graph.invoke(None, config)
```

### 在节点内读取 drain 状态

通过 `runtime` 参数访问 drain 状态，在到达超级步骤边界之前调整节点行为：

```python
from langgraph.runtime import Runtime

async def my_node(state: State, runtime: Runtime) -> State:
    if runtime.drain_requested:
        # 跳过昂贵的工作并返回最小结果
        return {"status": "skipped", "reason": runtime.drain_reason}
    return {"status": await do_work()}
```

### SIGTERM 钩子模式

处理进程关闭的推荐模式：

```python
import signal
from langgraph.runtime import RunControl
from langgraph.errors import GraphDrained

control = RunControl()
signal.signal(signal.SIGTERM, lambda *_: control.request_drain("sigterm"))

try:
    result = graph.invoke(inputs, config, control=control)
except GraphDrained as e:
    log.info("graph drained: %s", e.reason)
    # 在下次启动时使用相同配置恢复
```

> `request_drain()` 不会取消正在运行的 asyncio 任务或终止线程。对于硬性上限，将 drain 与优雅超时和任务取消配对使用。
