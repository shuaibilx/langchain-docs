# Voice Agent — 流程详细总结

## 两种架构对比

```
Sandwich 架构 (三明治)              Speech-to-Speech (S2S)
┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐   ┌─────┐ ┌─────────────┐ ┌─────┐
│ Mic │→│ STT │→│Agent│→│ TTS │→  │ Mic │→│ 多模态模型   │→│Spkr │
│     │ │     │ │     │ │     │ │  │     │ │             │ │     │
└─────┘ └─────┘ └─────┘ └─────┘   └─────┘ └─────────────┘ └─────┘
  音频   文本    文本    文本  音频   音频      直接处理      音频

3个独立组件, 可替换                 1个模型, 端到端
延迟: ~700ms                       延迟: 更低
可控性: 高                         可控性: 低
```

## Sandwich 架构详解

### 三步管道

```
浏览器麦克风
    │ PCM 音频流
    ▼
┌─────────────────────────────────────┐
│ Step 1: STT (AssemblyAI)            │
│                                     │
│  生产者: 发送音频块到 STT 服务        │
│  消费者: 接收转录事件                │
│                                     │
│  事件类型:                           │
│  ├── stt_chunk: 部分转录 (实时)      │
│  └── stt_output: 最终转录 (触发代理) │
└──────────────┬──────────────────────┘
               │ 文本转录
               ▼
┌─────────────────────────────────────┐
│ Step 2: LangChain Agent             │
│                                     │
│  输入: HumanMessage(transcript)      │
│  输出: 流式文本响应                   │
│                                     │
│  特性:                               │
│  ├── stream_mode="messages" 实时输出 │
│  ├── checkpointer 维护对话上下文     │
│  └── 自定义工具 (点餐等)             │
└──────────────┬──────────────────────┘
               │ 文本 chunk
               ▼
┌─────────────────────────────────────┐
│ Step 3: TTS (Cartesia)              │
│                                     │
│  收到文本即开始合成                   │
│  流式返回 PCM 音频块                 │
└──────────────┬──────────────────────┘
               │ PCM 音频流
               ▼
          浏览器扬声器播放
```

### 关键技术点

#### 1. Producer-Consumer 模式

```python
# STT 和 TTS 都使用这个模式
async def send_audio():        # 生产者：发送数据
    async for chunk in audio:
        await service.send(chunk)

async for event in receive():  # 消费者：接收结果
    yield event
```

两个任务通过 `asyncio.create_task` 并发运行。

#### 2. 流式处理

```python
pipeline = (
    RunnableGenerator(stt_stream)      # Audio → STT events
    | RunnableGenerator(agent_stream)  # STT events → Agent events
    | RunnableGenerator(tts_stream)    # Agent events → TTS audio
)
```

- 每个阶段独立并发
- 下游无需等上游全部完成
- 实现低延迟（<700ms）

#### 3. 事件流

```
事件类型传递链:
audio_bytes → stt_chunk → stt_output → agent_chunk → tts_chunk
    │            │            │             │            │
    ▼            ▼            ▼             ▼            ▼
  原始音频    部分转录     最终转录      代理响应      合成音频
```

所有上游事件会**透传**到下游，允许客户端追踪完整管道状态。

### WebSocket 通信

```python
# 客户端 → 服务端: 音频流
@app.websocket("/ws")
async def websocket_endpoint(websocket):
    async def audio_stream():
        while True:
            data = await websocket.receive_bytes()  # 接收音频块
            yield data

    # 服务端 → 客户端: 合成音频
    async for event in pipeline.atransform(audio_stream()):
        if event.type == "tts_chunk":
            await websocket.send_bytes(event.audio)  # 返回音频块
```

### Agent 配置

```python
agent = create_agent(
    model="google_genai:gemini-3.5-flash",
    tools=[add_to_order, confirm_order],
    system_prompt="Be concise. Do NOT use emojis.",  # TTS 不支持 emoji
    checkpointer=InMemorySaver(),  # 对话记忆
)
```

关键：**system_prompt 要求不使用 emoji 和特殊字符**，因为 TTS 引擎无法正确发音。

### 组件替换

| 组件 | 默认 | 可替换为 |
|------|------|---------|
| STT | AssemblyAI | Google Speech, Whisper, Deepgram |
| Agent | LangChain Agent | 任何 text-in/text-out 模型 |
| TTS | Cartesia | ElevenLabs, Azure TTS, Google TTS |
| 传输 | WebSocket | WebRTC, Twilio, Vonage |

## 性能优化

| 优化点 | 方法 |
|--------|------|
| 降低延迟 | 流式处理，不等完整响应 |
| 并发执行 | asyncio.create_task 并行 |
| 选择模型 | gemini-3.5-flash（快）而非 gpt-4o（慢） |
| 音频格式 | PCM 原始格式，减少编解码开销 |
| 连接复用 | WebSocket 长连接，避免重复握手 |
