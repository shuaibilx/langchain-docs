## 四个阶段详解

### Phase 1: Classify / 分类

```python
structured_llm = router_llm.with_structured_output(ClassificationResult)
result = structured_llm.invoke([
    {"role": "system", "content": "分析查询，决定调用哪些知识库..."},
    {"role": "user", "content": state["query"]}
])
```

| 技术 | 作用 |
|------|------|
| `with_structured_output` | 强制 LLM 输出 Pydantic 模型 |
| `ClassificationResult` | 包含 `classifications: list[Classification]` |
| 每个 Classification | `source` + 针对性 `query` |

**为什么用结构化输出？**
- 比自由 JSON 解析更可靠
- 路由逻辑显式且可测试
- 只返回相关源，不相关的被省略

### Phase 2: Route / 路由

```python
def route_to_agents(state: RouterState) -> list[Send]:
    return [Send(c["source"], {"query": c["query"]}) for c in state["classifications"]]
```

| 概念 | 说明 |
|------|------|
| `Send(node, data)` | 指定目标节点和传递的状态 |
| 并行执行 | 多个 Send 同时执行 |
| 精简输入 | 每个代理只收到 `AgentInput`（仅 query） |

### Phase 3: Query / 查询

各代理独立运行，互不干扰：

```python
def query_github(state: AgentInput) -> dict:
    result = github_agent.invoke({"messages": [{"role": "user", "content": state["query"]}]})
    return {"results": [{"source": "github", "result": result["messages"][-1].content}]}
```

关键：返回格式统一 `{source, result}`，通过 reducer 汇总。

### Phase 4: Synthesize / 综合

```python
def synthesize_results(state: RouterState) -> dict:
    formatted = [f"**From {r['source'].title()}:**\n{r['result']}" for r in state["results"]]
    response = router_llm.invoke([
        {"role": "system", "content": f"综合结果回答: {state['query']}"},
        {"role": "user", "content": "\n\n".join(formatted)}
    ])
    return {"final_answer": response.content}
```

## 关键技术点

### StateGraph 构建

```python
StateGraph(RouterState)
    .add_node("classify", classify_query)
    .add_node("github", query_github)
    .add_node("notion", query_notion)
    .add_node("slack", query_slack)
    .add_node("synthesize", synthesize_results)
    .add_edge(START, "classify")
    .add_conditional_edges("classify", route_to_agents, ["github", "notion", "slack"])
    .add_edge("github", "synthesize")
    .add_edge("notion", "synthesize")
    .add_edge("slack", "synthesize")
    .add_edge("synthesize", END)
    .compile()
```

| 方法 | 作用 |
|------|------|
| `add_node` | 添加节点（函数） |
| `add_edge` | 添加固定边 |
| `add_conditional_edges` | 添加条件边（通过 `route_to_agents` 返回 `Send`） |

### Reducer 机制

```python
results: Annotated[list[AgentOutput], operator.add]
```

- `operator.add` = 列表拼接
- 多个并行代理各自返回 `[{"source": "...", "result": "..."}]`
- Reducer 将它们合并为 `state["results"]`

### 与 Supervisor 模式的对比

| 维度 | Supervisor | Router |
|------|-----------|--------|
| 路由方式 | LLM 自主选择工具 | 结构化输出 + Send API |
| 执行方式 | 顺序工具调用 | 并行 Send |
| 控制粒度 | LLM 决定 | 代码显式控制 |
| 适用场景 | 动态决策 | 已知垂直领域 |

### 与 Subagents 模式的对比

| 维度 | Subagents | Router |
|------|----------|--------|
| 路由决策 | LLM 动态决定 | 结构化分类 |
| 执行方式 | 顺序 | 并行 |
| 预处理 | 无 | 生成针对性子问题 |
| 适用场景 | 不确定调用谁 | 已知知识源 |

## 有状态路由器

### 工具包装方式（推荐）

```
用户消息
    │
    ▼
Conversational Agent (有 checkpointer)
    │
    ├── tool_call: search_knowledge_base("query")
    │       │
    │       ▼
    │   Router (无状态)
    │       ├── Classify → Send
    │       ├── GitHub Agent
    │       ├── Notion Agent
    │       └── Synthesize → result
    │
    └── 基于结果 + 对话历史 → 回答
```

路由器保持无状态，对话代理处理记忆。用户可以多轮对话，代理按需调用路由器。
