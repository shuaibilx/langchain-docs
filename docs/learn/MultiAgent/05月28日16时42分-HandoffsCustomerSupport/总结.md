# Handoffs Customer Support — 流程详细总结

## 核心概念：状态机模式

```
┌──────────────────────────────────────────────────────┐
│                   同一个 Agent                        │
│                                                      │
│  ┌──────────────┐  ┌──────────────┐  ┌────────────┐ │
│  │ Config A     │  │ Config B     │  │ Config C   │ │
│  │              │  │              │  │            │ │
│  │ prompt: "问保修" │ │ prompt: "分类" │ │ prompt: "解决" │ │
│  │ tools: [     │  │ tools: [     │  │ tools: [   │ │
│  │  record_warr │  │  record_type │  │  solution  │ │
│  │ ]            │  │ ]            │  │  escalate  │ │
│  └──────┬───────┘  └──────┬───────┘  └─────┬──────┘ │
│         │                 │                │        │
│         ▼                 ▼                ▼        │
│  current_step:     current_step:    current_step:   │
│  warranty_collector  issue_classif  resolution_spec  │
└──────────────────────────────────────────────────────┘
```

**与子代理模式的区别：**
| 子代理模式 | 状态机模式 |
|-----------|-----------|
| 多个独立代理 | 单个代理，多个配置 |
| 子代理作为工具被调用 | 工具更新状态，中间件切换配置 |
| Supervisor 路由 | current_step 驱动 |

## 状态转换流程

```
Turn 1: 用户 "我的屏幕碎了"
    │
    ▼ current_step = "warranty_collector" (默认)
    │ 中间件注入: WARRANTY_COLLECTOR_PROMPT + [record_warranty_status]
    │
    ▼ Agent 回复: "请问您的设备还在保修期内吗？"

Turn 2: 用户 "是的，在保修期内"
    │
    ▼ Agent 调用: record_warranty_status("in_warranty")
    │   └── 返回 Command(update={warranty_status, current_step="issue_classifier"})
    │
    ▼ current_step = "issue_classifier"
    │ 中间件注入: ISSUE_CLASSIFIER_PROMPT + [record_issue_type]
    │
    ▼ Agent 回复: "请描述一下您的问题"

Turn 3: 用户 "屏幕物理碎裂"
    │
    ▼ Agent 调用: record_issue_type("hardware")
    │   └── 返回 Command(update={issue_type, current_step="resolution_specialist"})
    │
    ▼ current_step = "resolution_specialist"
    │ 中间件注入: RESOLUTION_SPECIALIST_PROMPT + [solution, escalate]
    │
    ▼ Agent: 硬件+在保 → provide_solution("保修维修流程...")

Turn 4: 用户 "我搞错了，设备过了保修期"
    │
    ▼ Agent 调用: go_back_to_warranty()
    │   └── 返回 Command(update={current_step="warranty_collector"})
    │
    ▼ 回到保修验证步骤
```

## 关键组件

### 1. 自定义状态 (SupportState)

```python
class SupportState(AgentState):
    current_step: NotRequired[SupportStep]     # 驱动状态机
    warranty_status: NotRequired[Literal[...]]  # 业务数据
    issue_type: NotRequired[Literal[...]]       # 业务数据
```

### 2. 工具返回 Command

```python
@tool
def record_warranty_status(status, runtime) -> Command:
    return Command(
        update={
            "messages": [ToolMessage(...)],
            "warranty_status": status,
            "current_step": "issue_classifier",  # ← 状态转换
        }
    )
```

关键：**工具不只是返回字符串，而是返回 Command 对象来更新状态。**

### 3. @wrap_model_call 中间件

```python
@wrap_model_call
def apply_step_config(request, handler):
    current_step = request.state.get("current_step", "warranty_collector")
    config = STEP_CONFIG[current_step]
    
    request = request.override(
        system_prompt=config["prompt"].format(**request.state),
        tools=config["tools"],
    )
    return handler(request)
```

每轮调用 LLM 前，中间件：
1. 读取 `current_step`
2. 查找对应配置
3. 用状态值格式化提示词
4. 注入提示词和工具

### 4. STEP_CONFIG 字典

```python
STEP_CONFIG = {
    "warranty_collector": {
        "prompt": "...",
        "tools": [record_warranty_status],
        "requires": [],                    # 无前置依赖
    },
    "issue_classifier": {
        "prompt": "...{warranty_status}...",
        "tools": [record_issue_type],
        "requires": ["warranty_status"],   # 需要保修状态
    },
    "resolution_specialist": {
        "prompt": "...{warranty_status}...{issue_type}...",
        "tools": [provide_solution, escalate_to_human],
        "requires": ["warranty_status", "issue_type"],
    },
}
```

## 三种状态转换方式

| 方式 | 触发者 | 示例 |
|------|--------|------|
| 工具返回 Command | Agent 主动调用工具 | record_warranty_status |
| 用户纠正 | 用户说"搞错了" | Agent 调用 go_back_to_* |
| 外部事件 | API 返回结果 | 外部系统触发状态变更 |

## 信息流

```
用户消息
    │
    ▼
Agent State:
    ├── messages: [...]
    ├── current_step: "issue_classifier"  ← 中间件读取
    ├── warranty_status: "in_warranty"    ← 工具写入
    └── issue_type: (待填)
    
    │ 中间件处理
    ▼
LLM 调用:
    ├── system_prompt: ISSUE_CLASSIFIER_PROMPT.format(warranty_status="in_warranty")
    └── tools: [record_issue_type]
    
    │ LLM 响应
    ▼
Tool Call: record_issue_type("hardware")
    │
    ▼ Command(update={...})
    
Agent State 更新:
    ├── current_step: "resolution_specialist"  ← 下轮中间件读取
    ├── warranty_status: "in_warranty"
    └── issue_type: "hardware"                 ← 新增
```

## 与 Supervisor 模式的对比

| 维度 | Supervisor 模式 | 状态机模式 (Handoffs) |
|------|----------------|---------------------|
| 代理数量 | 多个（supervisor + workers） | 单个 |
| 行为变化 | 路由到不同子代理 | 动态切换配置 |
| 状态管理 | 各子代理独立 | 统一 state |
| 工具作用 | 调用子代理 | 更新状态 + 业务操作 |
| 适用场景 | 多领域并行 | 顺序流程 |

## 回退机制

```python
@tool
def go_back_to_warranty() -> Command:
    return Command(update={"current_step": "warranty_collector"})

@tool
def go_back_to_classification() -> Command:
    return Command(update={"current_step": "issue_classifier"})
```

只在 resolution_specialist 步骤提供回退，因为：
- 保修验证 → 不能回退（第一步）
- 问题分类 → 可以回退到保修验证
- 解决方案 → 可以回退到前两步
