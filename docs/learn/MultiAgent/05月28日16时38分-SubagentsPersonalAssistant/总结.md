## 数据流详解

### 单领域请求

```
用户: "Schedule a team standup tomorrow at 9am"
    │
    ▼ Supervisor
    │  LLM 判断: 这是日历任务
    │
    ├── tool_call: schedule_event("team standup tomorrow at 9am")
    │       │
    │       ▼ Calendar Agent (独立上下文)
    │       │  LLM 解析: "tomorrow at 9am" → "2024-06-19T09:00:00"
    │       │
    │       ├── tool_call: get_available_time_slots([], "2024-06-19", 60)
    │       │   → ["09:00", "14:00", "16:00"]
    │       │
    │       ├── tool_call: create_calendar_event("Team Standup", "2024-06-19T09:00:00", ...)
    │       │   → "Event created: ..."
    │       │
    │       └── 返回: "The team standup is scheduled for tomorrow at 9:00 AM"
    │
    └── Supervisor 综合结果 → 最终回复
```

### 跨领域请求（并行调用）

```
用户: "Schedule meeting with design team Tuesday 2pm, send email reminder"
    │
    ▼ Supervisor
    │  LLM 判断: 需要日历 + 邮件两个操作
    │
    ├── tool_call: schedule_event("meeting with design team Tuesday 2pm")
    │   └── Calendar Agent → get_available → create_event → 确认
    │
    ├── tool_call: manage_email("send design team reminder about mockups")
    │   └── Email Agent → send_email → 确认
    │
    └── Supervisor 综合两个结果 → 最终回复
```

## 工具包装模式

### 关键设计：子代理 → 工具

```python
@tool
def schedule_event(request: str) -> str:
    """描述帮助 supervisor 决定何时使用此工具"""
    result = calendar_agent.invoke(...)  # 子代理独立运行
    return result["messages"][-1].text   # 只返回最终文本
```

| 层级 | 看到的工具 | 决策类型 |
|------|-----------|---------|
| Supervisor | `schedule_event`, `manage_email` | 路由到哪个领域 |
| Calendar Agent | `create_calendar_event`, `get_available_time_slots` | 具体 API 调用 |
| Email Agent | `send_email` | 具体 API 调用 |

**为什么只返回最终文本？**
- Supervisor 不需要看到子代理的中间推理
- 减少 supervisor 的上下文消耗
- 保持各层关注点分离

## Human-in-the-loop 实现

### 中断点配置

```
Supervisor (checkpointer 在这里)
    │
    ├── schedule_event → Calendar Agent
    │       └── create_calendar_event ← ⏸️ 中断点 1
    │
    └── manage_email → Email Agent
            └── send_email ← ⏸️ 中断点 2
```

**关键规则：checkpointer 只加在顶层 supervisor**
- 子代理不需要 checkpointer
- 中断和恢复由 supervisor 统一管理

### 三种恢复方式

```python
# 1. 批准 — 执行原始操作
Command(resume={id: {"decisions": [{"type": "approve"}]}})

# 2. 编辑 — 修改参数后执行
edited = interrupt.value["action_requests"][0].copy()
edited["args"]["subject"] = "New subject"
Command(resume={id: {"decisions": [{"type": "edit", "edited_action": edited}]}})

# 3. 拒绝 — 跳过此操作
Command(resume={id: {"decisions": [{"type": "reject"}]}})
```

### 多中断点处理

```python
# 收集所有中断
interrupts = []
for step in supervisor_agent.stream(..., config):
    if "__interrupt__" in step:
        interrupts.append(step["__interrupt__"][0])

# 为每个中断指定不同决策
resume = {}
for interrupt in interrupts:
    if "email" in interrupt.value["action_requests"][0]["description"]:
        resume[interrupt.id] = {"decisions": [{"type": "reject"}]}
    else:
        resume[interrupt.id] = {"decisions": [{"type": "approve"}]}

# 一次性恢复所有
agent.stream(Command(resume=resume), config)
```

## 信息流控制

### 默认信息流

```
Supervisor → 子代理: 只传 request 字符串
子代理 → Supervisor: 只传最终文本
```

### 增强信息流（传完整上下文）

```python
@tool
def schedule_event(request: str, runtime: ToolRuntime) -> str:
    # 从 supervisor 的 state 取原始用户消息
    original = next(m for m in runtime.state["messages"] if m.type == "human")
    prompt = f"User inquiry: {original.text}\n\nSub-request: {request}"
    result = calendar_agent.invoke({"messages": [{"role": "user", "content": prompt}]})
    return result["messages"][-1].text
```

用途：解决 "schedule it for the same time tomorrow" 这类需要上下文的歧义请求。

## 与 DeepAgent 的对比

| 特征 | LangChain Subagents | DeepAgent Subagents |
|------|--------------------|--------------------|
| 定义方式 | 手动创建 agent + @tool 包装 | `subagents=[{...}]` 配置 |
| 工具传递 | 显式在 @tool 中调用 | 自动委派 |
| 上下文隔离 | 子代理独立 invoke | 子代理独立上下文 |
| 人在回路 | HumanInTheLoopMiddleware | 同 |
| 信息流 | 需手动控制 | 自动 |

## 设计原则

### 何时用 Supervisor 模式

```
✓ 多个不同领域（日历、邮件、CRM、DB）
✓ 每个领域有多个工具或复杂逻辑
✓ 需要集中式工作流控制
✓ 子代理不需要直接与用户对话
```

### 何时不用

```
✗ 只有几个工具 → 单个代理即可
✗ 需要代理与用户直接对话 → 用 handoffs
✗ 代理间需要对等协作 → 其他多代理模式
```
